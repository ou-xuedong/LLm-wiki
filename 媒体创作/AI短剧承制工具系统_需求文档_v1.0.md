## 一、项目背景与目标

### 1.1 根本诉求

杭州是之文化是一家 AI 短剧承制公司（AI 真人短剧 + AI 漫剧），成立于 2024 年，当前从试制迈向量产阶段。公司已串通一条 AI 短剧制作流程，但这条流程是"不稳定的人肉流水线"——不可控、不可复制、产能瓶颈、无法 24 小时生产。

核心困境用一句话说：**用一个高价不稳定的视频生成模型（Seedance 2.0），靠人的经验和体力去填补所有上游决策的不确定性。**

外部顾问 Owen 指出的方向是"把昂贵的视频抽卡前置工作做到极致"，即用极廉价的文字大模型把决策性工作全部消化掉，再触发昂贵的视频生成。这是本工具系统的核心方法论。

### 1.2 量化目标

| 目标 | 当前 | 目标 | 路径 |
|---|---|---|---|
| 日产能 | 1 集/人/天（一般要求） | 2 集/人/天 | 工具替代+夜间机器跑 |
| 平均抽卡次数 | 2.5 次/镜头 | 1.5 次/镜头 | 抽卡前准备极致化 |
| 夜间算力 | 完全闲置（12h浪费） | 无人值守批量抽卡 | 任务队列+异步调度 |
| 新人上手 | 靠老员工带（慢） | 3-5 天可上手 | SOP 工具化+样本库 |
| 生产品类头部 | 暂无 | 某个细分品类做到头部 | 聚焦+数据积累 |

## 三、当前制作工作流

### 3.1 流程总览（承制视角，从拿到剧本起）

```
剧本筹备（平台/IP方负责，不在我们工作范围）
        ↓
② 剧资生产：Nano Banana 2 / 即梦 / 豆包 → 角色图 + 场景图
        ↓
③ 分镜头脚本：组长用 Gemini 辅助，半手工写分镜表（30-60个镜头/集）
        ↓ ⚠️ 瓶颈
④ 任务拆分 + 抽卡：组员在 LibTV 调用 Seedance 2.0，生成 2-8 秒视频片段
        ↓ ⚠️ 瓶颈（最大）
⑤ 剪辑合成：剪映 / Premiere → 2-3 分钟成片 .mp4
        ↓ ⚠️ 瓶颈
⑥ 交付（平台/客户）
```

### 3.2 各环节现状

**② 剧资生产**
- 输入：剧本中描述的角色、场景
- 产出：角色图（每角色 3-5 张种子图）+ 场景图（每场景 1-2 张）
- 工具：Nano Banana 2（主力，多图融合）、即梦（中式审美）、豆包（快速概念图）
- 耗时：0.5-1 天/集
- 痛点：美术依赖个人审美，组员间风格难统一

**③ 分镜头脚本（瓶颈）**
- 输入：剧本 + 剧资
- 产出：分镜表（Excel/飞书），通常 30-60 个镜头/集
- 工具：Gemini 辅助 + 组长经验
- 耗时：0.5-1 天/集（组长 50% 时间花在这里）
- 痛点：组长是核心稀缺角色，3 个组需要 3 个强组长；组长水平参差导致后续抽卡崩溃

**④ 任务拆分 + 抽卡（最大瓶颈）**
- 输入：分镜表 + 剧资
- 产出：每镜头 2-8 秒视频
- 工具：LibTV → Seedance 2.0（主力视频生成模型）
- 耗时：1-2 天/集，难抽镜头卡更久
- 痛点：
  - 抽卡是算力消耗大头（Seedance 2.0，¥0.35/秒，3秒镜头 ¥1+）
  - 同镜头有人 1 次成，有人 8 次崩
  - 成功经验留在个人脑子里，无法在团队复制
  - 角色一致性/构图一致性/风格一致性靠人工目测，经常崩

**⑤ 剪辑合成（瓶颈）**
- 输入：所有抽卡产出的小视频
- 产出：2-3 分钟成片 .mp4
- 工具：剪映 / Adobe Premiere
- 耗时：0.5-1 天/集
- 痛点：剪辑发现镜头接不上 → 打回组员重抽 → 又一轮成本

### 3.3 一集成本结构

| 成本项 | 跑量剧 | 精品剧 |
|---|---|---|
| 纯 AI 算力（剧资+抽卡） | ¥200-500 | ¥1000-2000 |
| 组长人力 | ¥300-500 | ¥500-800 |
| 组员人力（4-6人） | ¥600-1200 | ¥1500-3000 |
| 剪辑人力 | ¥200-400 | ¥500-1000 |
| **合计** | **¥1300-2600** | **¥3500-6800** |

> **关键洞察**：无论跑量剧还是精品剧，人力成本始终占 70-80%。工具的核心价值不只是"省 AI 钱"，更是"解放组长和组员，让他们做更多集/做更精的集"。

## 四、核心痛点分析（工程语言）

### 痛点 A：非确定性生成，无可观察性

- **业务说**：同样的提示词、同样的首帧图，在 Seedance 2.0 上跑两次，结果差异大。失败时不知道为何失败，只能重试。
- **技术翻译**：生成式 API 的 stochastic 特性 + 黑盒模型，导致缺乏可观察性。Failure mode 不可枚举、不可定位。这种 API 在工程上属于"有损调用"，必须配套重试 + 选优 + 监控机制。
- **应对策略**：
  1. 无法消除非确定性，但可以压缩到最小范围——把所有可控因素（角色、场景、构图、首帧、提示词）固化到极致，只把"运动"这一个变量留给 Seedance
  2. 建立失败模式分类（脸崩、肢体崩、动作变形...），作为 Prompt 优化的反馈信号

### 痛点 B：成功经验不可复制（无样本沉淀机制）

- **业务说**：组员小王今天抽到特别好的镜头，用了某个绝妙提示词，但自己也说不清为什么行。第二天小李遇到类似镜头，只能从零摸索。
- **技术翻译**：缺乏 Experience Replay 机制。每个 API 调用是独立事件，成功路径没有被结构化记录，无法形成数据资产。每个员工都在本地训练自己的"私有模型"，组织没有积累。
- **应对策略**：
  1. "成功样本库"：每个好镜头自动记录提示词、参数、首帧、抽卡次数、负责人
  2. 新镜头要抽时，先在样本库里找相似镜头的成功 Prompt 作为参考
  3. 长期："Prompt 模式提取"——从大量成功样本里反向归纳出结构化的"Prompt 模板"

### 痛点 C：跨镜头一致性（角色/场景/风格）

- **业务说**：男主在第 5 个镜头脸变了，女主领口从 V 领变圆领，客厅色调从暖橙变冷蓝。这种崩溃在剪辑环节才被发现，整集要返工。
- **技术翻译**：跨独立 API 调用的状态一致性问题。本质是 Stateless API 上构建 Stateful 应用的经典挑战。需要外部状态存储（角色档案库）+ 强制注入（参考图/LoRA/主体库）+ 验证（产出后比对）+ 回退（崩了就重抽）。
- **应对策略**：
  1. 每个角色在 LibTV 主体库训练一个"主体"，所有镜头强制引用
  2. 每个场景第一次生成后存为"场景母版"，后续镜头都从母版的角度参考
  3. 剧资环节升级为"视觉规范手册"，不只是几张图，而是一份所有人遵守的规范

### 痛点 D：组长是核心瓶颈，稀缺且难培养

- **业务说**：3 个组就要 3 个组长。组长既要懂剧、懂分镜、懂 AI 工具脾气、还要能管 4-6 个人，这种人极其稀缺，招聘和培养都慢。
- **技术翻译**：系统对"高级人工节点"的依赖度过高。组长承担了"剧本→分镜"的转换函数，这个函数当前只能由人执行。这是系统扩展性的硬约束——想从 3 集/月扩到 10 集/月，要么招到 9 个组长（几乎不可能），要么把组长的部分能力工具化。
- **应对策略**：
  1. 组长的工作可以拆成两类：**判断类**（这个钩子够不够强、这个镜头是否合理）和**执行类**（把剧本写成分镜表）
  2. "执行类"应该被工具替代，组长只需做"判断类"

### 痛点 E：夜间生产空窗，资源浪费

- **业务说**：组员晚上 7 点下班，LibTV 服务器闲着，Seedance 2.0 算力闲着，但抽卡这事得人在电脑前一直点。一晚上 12 小时，50% 的潜在产能被浪费。
- **技术翻译**：缺乏异步批处理 + 无人值守能力。当前的抽卡流程是同步交互式的，没有任务队列、调度器、状态机。
- **应对策略**：白天人工准备（剧本/分镜/首帧），夜间机器抽卡（自动调用 LibTV API + 重试 + 监控）

## 五、"抽卡前准备"方法论

### 5.1 Owen 的核心洞察

> "我们要善于运用相对廉价的文字大模型，优化我们的抽卡前准备。也就是说，在抽卡（大量消耗 token）之前，要把准备工作优化到极致——这需要成熟的方法论。"

用程序员语言翻译：这是一个"把昂贵操作 cache 化、把便宜操作前置化"的工程思维。

| 类型 | 调用成本 | 不确定性 | 失败代价 |
|---|---|---|---|
| 文字大模型（GPT-4o/Claude/Gemini/DeepSeek） | 极低（$0.001-0.015/千token） | 低（可重复、可校验） | 极低（失败重跑成本几分钱） |
| 视频大模型（Seedance 2.0） | 高（¥0.35/秒，3秒镜头 ¥1+） | 高（同输入不同输出） | 高（一次失败=1-2元+时间） |

**关键认知**：文字模型的不确定性可以通过多次调用 + 投票/校验快速消除，几乎不增加成本。视频模型的不确定性消除一次就要花真金白银。所以理性的工程做法，是把所有"决策性工作"全部前置到文字模型，把视频模型当"哑执行器"用。

### 5.2 三阶段工作流

```
📝 文字模型阶段（廉价 + 多轮迭代，可承担反复调用）
  把所有可以用文字模型解决的事情都解决：
  分镜拆分 / 提示词生成 / 提示词校验 / 提示词翻译 / 失败模式预测 / 降级方案生成

🎨 生图模型阶段（中等成本）
  把所有可以用生图模型解决的事情都解决：
  角色定档 / 场景母版 / 首帧图（多张候选）/ 一致性校验

🎬 视频模型阶段（昂贵 + 不确定，只调用一次）
  每个镜头喂给 Seedance 2.0 时：
  ✓ 所有决策已固化
  ✓ 所有素材已就位
  ✓ 所有提示词已优化
  ✓ 所有失败方案已备好
  → 视频模型只剩"让首帧动起来"这一件事
```

### 5.3 同一镜头的两种做法对比

❌ **当前做法（组员经验式）**
1. 组员看分镜表，大致理解
2. 自己想首帧，在 Nano Banana 2 抽几张
3. 选一张觉得不错的
4. 自己写提示词（"男主走出门"）
5. 喂 Seedance，失败
6. 改提示词重抽，失败
7. 改首帧重抽，失败
8. 改提示词，可能成功
9. **共消耗 5-10 次抽卡**

✅ **方法论式做法**
1. 分镜工具自动产出标准化首帧描述 + Seedance 提示词（文字模型）
2. 提示词经过校验子模型评分（文字模型）
3. 系统自动从样本库找相似成功镜头的提示词作为参考（向量检索）
4. 生图模型按规范产出 3 张候选首帧，组员选最佳
5. 抽卡工具把首帧+优化后提示词喂 Seedance
6. **大概率 1-2 次成，共消耗 1-2 次抽卡**

## 六、工具系统需求设计

### 6.1 系统定位

本系统定位为**AI 短剧承制的工程化工具链**，覆盖从"分镜"到"成片"的核心环节，不做剧本生成（承制方不碰剧本），目标是：

1. 把组长从"执行类"工作中解放出来，专注"判断类"工作
2. 把组员从"试错式抽卡"转变为"有的放矢的精准抽卡"
3. 实现夜间无人值守批量生产
4. 把个人经验转化为组织资产（样本库）

### 6.2 模块划分

#### 模块一：分镜工具（Storyboard Generator）

**输入**：Word 剧本（.docx）或纯文本

**输出**：
- 结构化分镜表（JSON 格式，含：镜头序号、景别、运镜、动作、台词、时长估算）
- 每个镜头的 Seedance 提示词（初版）
- 每个镜头的首帧描述文本

**核心逻辑**：
1. Stage 0：剧本 → 结构化 JSON（角色列表、场景列表、分场）
2. Stage 1：注入"爆款基因"战略（参考成功样本库中同题材/同类型的爆款分镜结构）
3. Stage 2：逐场生成分镜镜头（每镜头输出：景别+运镜+动作描述+Seedance 提示词）
4. Stage 3：全局校验（检查一致性：角色名统一、场景名统一、时长逻辑）

**成本控制**：全程使用文字大模型，可反复迭代，成本极低

**关键约束**：不追求一次做对，追求快速迭代；设置最大迭代次数防止死循环

#### 模块二：首帧候选工具（First Frame Generator）

**输入**：分镜表（来自模块一）+ 角色设定 + 场景设定

**输出**：每个镜头的 3 张候选首帧图（由生图模型产出）

**核心逻辑**：
1. 文字模型根据分镜表生成每张首帧的画面描述（Prompt）
2. 生图模型（Nano Banana 2 / 即梦）产出 3 张候选图
3. 文字模型对 3 张图做质量预筛（构图、角色脸、风格一致性）
4. 组员从 3 张中选 1 张（或打回重抽）

**成本控制**：生图成本中等，但每镜头只抽 3 张（而非无限制抽），文字模型做预筛减少无效生图

#### 模块三：抽卡工具（抽卡自动化 + 夜间无人值守）

**输入**：分镜表 + 首帧图 + 优化后的提示词

**输出**：每个镜头的视频文件（2-8 秒）+ 状态回写（成功/失败/原因标注）

**核心逻辑**：
1. 任务队列：把待抽卡镜头按优先级排队
2. 批量提交：5-10 并发调用 LibTV API（Seedance 2.0）
3. 轮询状态：定期查询任务完成情况
4. 自动重试：失败的镜头按策略自动重试（更换提示词/更换首帧/保持原样）
5. 选优：每个镜头生成 N 个版本，剪辑师/组员选最优
6. 夜间模式：白天提交任务，夜间自动跑，次日早上出日报

**关键约束**：
- 需要处理 LibTV API 的鉴权、限流、超时
- 需要处理网络异常（国内访问 LibTV 可能涉及部分国际接口）
- 需要分段排错能力（飞书+LibTV+视频下载，整条链路哪个环节坏要知道）

#### 模块四：样本沉淀器（Experience Replay System）

**输入**：每次抽卡的结果（成功 or 失败 + 原因）

**输出**：可查询的成功样本库（按场景/角色/镜头类型索引）

**核心逻辑**：
1. 每次抽卡完成后，自动记录：
   - 剧本信息（题材、类型）
   - 角色信息（角色名、类型）
   - 场景信息（场景名、类型）
   - 镜头信息（景别、运镜、动作类型）
   - 提示词（最终使用的那个）
   - 首帧图（URL）
   - 抽卡次数
   - 结果（成功/失败+失败原因分类）
   - 负责人
2. 向量检索：新镜头要抽时，根据镜头特征找相似成功样本作为参考
3. 统计面板：展示各类型镜头的平均抽卡次数、成功率、常见失败模式

### 6.3 一体化 vs 模块化（待确认的核心问题）

> ⚠️ **这是我们最想和工程伙伴讨论的核心问题之一**

工具应该按"廉价文字模型 → 中等生图模型 → 昂贵视频模型"的成本梯度拆成独立模块，还是合成为一个一体化 Pipeline？

**分立模块的优点**：
- 各模块可独立迭代
- 失败隔离，单模块挂不影响其他
- 可以让组长/组员只在需要的模块介入

**一体化 Pipeline 的优点**：
- 数据流连续，减少人工干预点
- 样本数据闭环，feedback 循环自动流转
- 用户体验简单（一个入口，一个出口）

**当前倾向**：先做模块化，核心数据流打通后逐步一体化。优先跑通模块一（分镜工具）和模块三（抽卡工具），因为这两头是当前最大瓶颈。

## 九、失败教训（避免重复踩坑）

### 9.1 分镜工具试做失败

**现场**：用 Claude API + Python 试做，四个 Stage 设计（剧本→JSON→注入爆款基因→逐场分镜→全局校验），消耗约 30 美金测试费后卡在测试循环，没有产出可用分镜表。

**可能原因**（不确定，待工程伙伴诊断）：
1. Stage 之间编排问题：无正确终止条件，陷入死循环
2. Prompt 设计问题：Claude 没理解任务边界，反复输出无效内容
3. JSON 解析失败导致重试耗尽
4. 模型选错：用了 Opus 而不是 Sonnet，token 单价高
5. 测试方法错：在沙箱里跑测试反复调用真实 API

### 9.2 抽卡工具试做失败

**现场**：用 Claude Code 试做 LibTV API 自动化（飞书拉取→LibTV请求→批量提交→轮询→下载→回写飞书），开发到最后一直网络报错，没有跑通。

**可能原因**：
1. LibTV API 鉴权/限流/超时配置不对
2. 网络代理问题（国内访问 LibTV 可能涉及部分国际接口）
3. LibTV API 文档没读全，参数传错
4. 整条调用链路太长（飞书+LibTV+视频下载），缺乏分段排错能力

### 9.3 认知更新

1. **方向都是对的**：工具协同的设计逻辑没有问题，业内头部公司都在做类似的事
2. **"业务方撸代码"是死路**：Prompt 工程、错误处理、成本控制、API 集成，每项都需要专业工程师
3. **最贵的不是开发费，是试错时间**：在错误方向上多花一周，就是 3 个组的产能损失
4. **下一步要先论证再开发**：必须先和工程伙伴充分讨论方案的合理性，再动手

## 十、实施路线图（建议）

### Phase 1：论证与原型（4-6 周）

**目标**：验证核心假设，确认技术方案可行

**内容**：
1. 工程伙伴深度研究 LibTV API（鉴权、限流、端点、参数）
2. 研究 Nano Banana 2 / 即梦的 API 接口
3. 用最小成本跑通"分镜工具"POC（只跑 Stage 0 → Stage 2，不做全局校验）
4. 用最小成本跑通"抽卡工具"POC（单镜头手动触发 → 自动轮询）
5. 产出：技术方案评审 + 原型演示

**交付**：技术方案文档 + 原型 Demo

### Phase 2：核心模块开发（8-12 周）

**目标**：把最大瓶颈（抽卡工具 + 分镜工具）做到生产可用

**内容**：
1. 分镜工具 v1.0（剧本 → 分镜表 + 提示词）
2. 抽卡工具 v1.0（夜间无人值守版本）
3. 基础样本沉淀（手动录入成功案例，v1.0 不做自动沉淀）
4. 飞书打通（任务推送 + 状态回写）

**交付**：可在生产环境使用的两个核心工具

### Phase 3：工具链串联（4-6 周）

**目标**：让数据流在模块之间流动，实现一体化体验

**内容**：
1. 首帧候选工具接入
2. 样本库自动沉淀（每次抽卡完成自动写入）
3. 分镜工具 → 首帧工具 → 抽卡工具的数据流串联
4. 向量检索功能上线（相似镜头推荐）

**交付**：完整工具链，数据闭环

### Phase 4：工业化固化（持续）

**目标**：形成可传承的 SOP，新人 3-5 天上手

**内容**：
1. Prompt 模板库建设
2. 失败模式分类体系完善
3. 新人培训文档和工具引导
4. 数据分析面板（产能、成功率、瓶颈定位）

## 十一、风险与约束

### 11.1 外部依赖风险

- **LibTV API 稳定性**：所有抽卡能力依赖 LibTV，若 LibTV 调整 API 或定价，直接影响系统
- **Seedance 2.0 模型迭代**：视频生成质量持续提升，但也可能带来新的不兼容性
- **平台政策变化**：红果等平台的内容政策、承制政策随时可能调整

### 11.2 内部约束

- **团队技术能力**：当前团队以影视为主，非技术背景；需要工程伙伴带路，不能做技术决策
- **现金流优先**：不烧钱扩张，工具投入需与现金流匹配
- **阶段目标明确**：先跑通"承制"这一段，不做剧本/投流/商业化延伸

### 11.3 待确认问题

1. LibTV API 的详细技术规格（需要实际测试）
2. Nano Banana 2 / 即梦的 API 接口是否支持自动化调用
3. 文字大模型选型（Claude / GPT-4o / Gemini / DeepSeek，哪家最适合分镜任务）
4. 向量数据库选型（需要存储和检索大量样本）
5. 服务器部署方案（国内访问 LibTV 的网络优化）
