## 1. 五段矩阵 S1→S5

| 段 | 名称 | 核心活动 | 主要产出 |

| S2 | 产品设计 | PRD + 用户旅程 + 原型 + 验收标准 | PRD_v0.x.md、用户旅程.md、prototype/、验收标准.md |

## 1.5 产出物格式约定（MD vs HTML · v1.0 · 2026-05-10）

> **核心判据**：洞察卡 `~/Desktop/AILab/知识库/wiki/insights/HTML优先输出范式_主人项目改造指引.md` §2 分层模型 + §3.5 四模式辨析。

任何产出物有两个生命阶段：

- 🟦 **协作期** → **MD**（频繁改 / git diff 友好 / Token 省 / Agent 实施）
- 🟧 **定稿期** → **生成 HTML 快照**（视觉化 / 评审 / 对外 / 可分享）

**铁律**：HTML 是"产物快照"，MD 是"工作底稿"。两者并存——定稿出 HTML，源 MD 永不删。

### 1.5.1 各阶段格式对照

| 阶段 | 协作期 MD（必有） | 定稿期 HTML（关键节点出快照） |
|---|---|---|
| S1 | 方向评估.md / 五看分析.md / 商业模式.md / 融资叙事.md / 方案/ / 讨论记录 / 决策日志 | S1→S2 跃迁评审 HTML（方向研判全景） |
| S2 | PRD_v0.x.md / 用户旅程.md / 验收标准.md | **PRD 定稿 HTML** / **用户旅程图 HTML**（视觉化是核心）/ prototype/ 本就是 HTML |
| S3 | code/ / Sprint/Sprint_YYYY-WW.md / debug 日志 / Code Review 记录 | Sprint 末评审 HTML / 架构图 HTML 快照 |
| S4 | qa-report.md / 发布清单.md / 监控Dashboard.md | 上线 GO/NO-GO 决策 HTML / 健康检查仪表盘 HTML |
| S5 | 周报/YYYY-WW.md / incident.md / 调研报告.md | 季度复盘 HTML / 重大 incident HTML（对外/老板） |

### 1.5.2 永远 MD（不出 HTML 快照）

- `讨论记录/` / `决策日志/`（流水档案，AI 检索 + 主人速读）
- `Sprint/`（Agent 要执行）
- `Raw文件/`（贴源不动）
- `_meta.yaml` / `README.md` / `CLAUDE.md`（工具协议层）
- 各 schema / 模板定义

### 1.5.3 触发 HTML 快照的时机

- ✅ 阶段跃迁评审（S1→S2 / S2→S3 / S3→S4 / S4→S5）
- ✅ 主人主动说"出报告 / 给 XX 看 / 做评审"
- ✅ 重大里程碑（首版 PRD 定稿 / Sprint 评审 / 上线 GO/NO-GO / 季度复盘）

### 1.5.4 视觉系统约定

首次需要 HTML 快照时：

- 在 `S2_产品设计/prototype/` 或项目 `模板/` 建 `design-system.html` 视觉基线
- 后续所有 HTML 快照统一引用该设计系统的 tokens
- 推荐栈：原生 HTML + Tailwind CDN + Chart.js CDN（与卟卟鸡同栈，零依赖、双击可开）

## 2. 项目目录架构

```
<项目>/
├── CLAUDE.md                  # 项目入口（顶部 @import 本文件）
├── README.md                  # 项目名片 + 阶段仪表盘
├── _meta.yaml                 # Agent 入口（产出契约 + 退出阈值）
├── 反馈日志.md                # 主人纠正/意见沉淀（按需新建）
├── Raw文件/                   # 原始资料贴源层（详见 §2.1）+ 内置 README.md
├── S1_方向探索/
│   ├── README.md              # 阶段说明 + 进入退出阈值 + 产出清单
│   ├── 方向评估.md             # 产出物（gstack 八问）
│   ├── 五看分析.md
│   ├── 商业模式.md
│   ├── 融资叙事.md
│   ├── 方案/                  # ≥3 份差异化方案
│   ├── 讨论记录/              # 流水账，按日期
│   └── 决策日志/              # 结构化，按主题
├── S2_产品设计/               # 同结构（PRD_v0.1.md / 用户旅程.md / 验收标准.md / prototype/）
├── S3_研发落地/               # 含 code/ + Sprint/
├── S4_上线交付/               # 含 发布清单 / GTM / 监控
└── S5_持续迭代/               # 含 BP / SP / 周报
```

**规则：**
- 每个 pending 阶段**预建骨架**（`README.md` 占位 + 子目录），不要等到激活才建
- 阶段目录三类共存：**产出物**（单文件或子目录）+ **讨论记录/**（按日期）+ **决策日志/**（按主题）
- 跨阶段讨论归属"当前主阶段"
- 阶段跃迁必须有决策日志作凭据，**禁止悄悄进阶**
- 根 `README.md` **只做三件事**：项目名片 / 阶段仪表盘 / 当前跃迁阈值，**不承载分析内容**

### 2.1 `Raw文件/` — 贴源层规范

`Raw文件/` 是项目的"贴源层"——保存外部输入、未经加工的原始资料。**类似数据仓库的 ODS（贴源层）**：进什么样、出什么样，不动一刀。

**该放：**

- 客户/受访者提供的原始文档（SOP、工作流、业务速览、数据样本）
- 访谈追问清单 / 录音转写
- 主人收集的第三方资料（行业报告、竞品截图、KOL 文章 PDF）
- 客户提供的"原版"文件——可保留原文件名以保真

**不该放：**

- 奴婢/Agent 整理后的产物 → 各 S1-S5 阶段的 `讨论记录/` `决策日志/`
- 跨项目通用实体卡片 → `~/Desktop/AILab/知识库/wiki/`（由知识库统一维护）
- 项目代码 → `S3_研发落地/code/`

**不可变原则：** Raw 是项目原始证据——

- 一进不修改，要改另存为新版本（`_v1` `_v2` `_终版`）
- 删除需主人确认
- 这条原则与"贴源层"语义等价，禁止破坏

**命名约定：**

- 已知来源 + 主题：`<来源>_<主题>_<版本>.<ext>`（投流范例：`sasa_访谈追问清单_v2.md`）
- 时间相关：`YYYY-MM-DD_<来源>_<主题>.<ext>`
- 客户原版可保留原文件名以保真

**敏感数据 / 大文件：**

- 真实账号、姓名、电话、API Key、攻击 payload 等入库前必须脱敏
- 单文件 > 10 MB 的音视频/数据集考虑 git-lfs 或外部存储链接（不直接入仓）

**README.md（每个项目必备）：** 每个项目的 `Raw文件/` 下放一个 `README.md`，含：本节通用规范引用 + 项目个性化"典型素材"示例 + 当前文件清单（按命名约定一行一个，或写"暂无原始素材，等待外部输入"）。

**适用范围：** 产品研发型项目（已毕业 + 孵化园未毕业方向；方向探索 CLAUDE.md §1 已联动说明）。工具运营型项目按各自领域规范，不强制 `Raw文件/`。

## 3. `_meta.yaml` 标准格式

每个产品研发型项目的 `_meta.yaml` 必须包含：

```yaml
project_name: 项目名称
state: 📐深度分析                 # 🔍初步探索 → 📐深度分析 → 🧪验证中 → ✅确定推进/❌放弃
strategy_mode: 守稳型             # 扩张型/精选扩张/守稳型/收缩型
current_stage: S1_方向探索        # 当前活跃阶段
started: YYYY-MM-DD

active_decision: 当前讨论中的决策（一句话）
last_action: 最近一次动作（日期 + 描述）
next_action: 下一步（待主人确认后执行）

pending_issues:                   # 悬而未决的关键问题
  - 问题1

# 阶段化产出契约与阈值（五段矩阵）
stages:
  S1_方向探索:
    status: active                # pending / active / done
    entered_at: YYYY-MM-DD
    required_deliverables:
      - { file: 方向评估.md, required: true, status: done }
      - { file: 五看分析.md, required: true, status: done }
      - { file: 商业模式.md, required: true, status: done }
      - { file: 融资叙事.md, required: false, status: done }
      - { file: 方案/, required: true, min_count: 3, status: done }
    exit_criteria:
      - { desc: gstack /office-hours 八问完成且无 🔴 低信心未决项, done: false, evidence: 方向评估.md§强制提问 }
      - { desc: 战略模式选定并写决策日志, done: false, evidence: 决策日志/战略模式.md }
  S2_产品设计:
    status: pending
    entry_criteria: inherit_from_S1_exit
    required_deliverables: [ ... ]
    exit_criteria: [ ... ]
  # S3_研发落地 / S4_上线交付 / S5_持续迭代 同结构

related_entities:                 # 来自知识库的相关实体
  - [[实体名]]

milestones:                        # 阶段演进里程碑
  - { date: YYYY-MM-DD, event: ... }
```

**Agent 工作流：**
1. 进项目 → 读 `_meta.yaml` → 根据 `current_stage` 定位到对应阶段目录
2. 读阶段 `README.md` → 理解产出清单、退出阈值、当前状态
3. 按 `active_decision` 和 `pending_issues` 推进讨论
4. 讨论有结论 → 写入**当前阶段**的 `决策日志/`（新建文件）
5. 同时更新 `_meta.yaml` 的 `last_action` / `next_action` / `stages.{当前}.exit_criteria` 勾选
6. 退出阈值全达成 → 写「阶段跃迁」决策日志 → 激活下一阶段 → 更新 `current_stage`
7. 到达 S5 且状态「✅确定推进」→ 触发知识库反向刷新（L1 §6.2）

## 5. 阶段跃迁守门铁律

`_meta.yaml.stages.{阶段}.exit_criteria` 中**必须包含至少一条 skill 产出凭据**作为退出条件。例：

```yaml
S1_方向探索:
  exit_criteria:
    - { desc: gstack /office-hours 八问全部完成且无低信心未决项, done: false, evidence: 方向评估.md }
    - { desc: gstack /plan-ceo-review 战略模式选定并写决策日志, done: false, evidence: 讨论记录/*战略模式.md }
    - { desc: 方案/ 下至少 3 份差异化方案，主人已选定主推, done: false, evidence: 决策日志/方案选定.md }
```

**没有 skill 产出凭据，禁止跃迁。** 这是端到端无断点的硬约束。

## 6. 工具缺失降级路径

| 缺失工具 | 阶段 | 降级方案 |
|---|---|---|
| superpowers | S1 候选方案发散 | `brainstorming` 改为"奴婢按方法论自行发散 ≥3 份方案"（方法论模式），不阻塞 |
| superpowers | S2 末（writing-plans） | **必须装**；提示主人安装；如主人明确拒绝，回退到"人工写 plan"，记入决策日志 |
| superpowers | S3 主线（TDD/调试/worktrees） | 同上，未装则 TDD 改为人工驱动，效率下降但流程等价 |
| gstack | 任何阶段 | **必须先装，无降级**——这是思维底座 |
| qmd | 任何阶段 | 提示主人安装；如拒绝，奴婢用 Read/Grep 直接读 `~/Desktop/AILab/知识库/wiki/` 替代 |

## 7. 阶段化工具重检触发点

L1 §9.1 启动自检是项目级总检。下列**阶段跃迁前**必须**再自检一次**对应工具：

|---|---|---|

## 8. 沉淀规范

- 每次阶段性结论，写入该阶段 `讨论记录/YYYY-MM-DD.md`
- 内容：讨论要点、做出的决策、待验证假设、下一步行动
- 同时更新该阶段 `README.md` 的"当前阶段"和"阶段结论"
- 所有讨论记录按日期归档，**不可覆盖只可追加**
- 放弃的方向保留完整记录和放弃原因，供未来复盘
