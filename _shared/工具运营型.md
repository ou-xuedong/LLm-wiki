# 工具运营型项目共享规范（L1.5）

> 本文件由工具运营型项目（知识库、卟卟鸡投资决策、autoResearch实验室）通过 L2 顶部 `@~/Desktop/AILab/_shared/工具运营型.md` 加载。
>
> 通用方法论（默认行为模式、gstack、20 条原则、知识库联动、反馈日志）见 `~/Desktop/AILab/CLAUDE.md`。本文件**仅承载工具运营型专属**——持续运行 + 决策辅助 + 信息维护类项目的共性规范。

## 1. 项目类型特征

| 维度 | 工具运营型 | 对照：产品研发型 |
|---|---|---|
| 主活动 | 信息维护 / 决策辅助 / 自动化任务 | 写代码做产品 |
| 流程 | 自定义领域工作流（如卟卟鸡决策四步、知识库摄入→消歧→编织） | 五段矩阵 S1→S5 |
| `_meta.yaml` | 不强制 | 必备 |
| 工具脚本 | 辅助手段，放 `工具/` 目录，不作产品分发 | 产品代码本身，放 `S3_研发落地/code/` |
| 状态 | 持续运行 | 阶段化推进 |
| superpowers | 不强求 | S2 末必装 |

## 2. 项目目录架构

```
<工具运营型项目>/
├── CLAUDE.md                  # 项目入口（顶部 @import 本文件）
├── README.md                  # 项目名片 + 当前位置
├── 反馈日志.md                # 主人纠正/意见沉淀（按需新建）
├── 工具/                      # 工具脚本（如 market_data.py）
├── <领域目录>/                # 领域专属（如卟卟鸡的 专家智库/、决策流程/、评估记录/）
└── log.md                     # 维护日志（机械追记 last_updated 等）
```

**规则：**
- 不强制五段矩阵骨架（`_meta.yaml`、`S1_xxx/` 等）
- 工具脚本统一放 `工具/` 目录，不要散落根目录
- 工具脚本变更要更新 `CLAUDE.md` 的工具清单
- 评估/操作记录按日期归档（如卟卟鸡 `评估记录/[YYYY-MM-DD]_[类型].md`），**不可覆盖只可追加**

## 3. 评估记录格式（持续运行的产物归档）

工具运营型项目每次执行任务都产出"一次的结果"（决策研判、信息摄入、自动化报告等），统一规范：

- **命名**：`<领域目录>/[YYYY-MM-DD]_[类型].md`
- **不可覆盖**：每次执行新建文件，不在原文件追加
- **可追加例外**：`log.md` 这类纯流水账文件可追加（按时间倒序）
- **回顾用途**：用于事后复盘、检验框架准确率、识别系统漂移

## 4. 持续运营 vs 升级改造

工具运营型项目两种工作模式：

| 模式 | 触发 | 流程 |
|---|---|---|
| **日常运营** | 主人触发既定操作（如卟卟鸡"市场体检"、知识库"摄入新情报"） | 按 L2 项目专属流程执行，无需多方案博弈 |
| **升级改造** | 涉及新工具/新流程/数据模型变更/新评估指标 | 套用 L1 §3 创造性写入流程：先议后定，多方案博弈，主人确认后执行；大改造可临时套用 S1 评估流程 |

**判定标准**：改动范围超过单个工作流的边界、引入新概念、影响其他项目接入方式 → 走改造模式。

## 5. 工具脚本管理

- 脚本统一放 `工具/` 目录
- 每个脚本顶部注释：用途、输入、输出、依赖
- 主要脚本在 `CLAUDE.md` 的"工具清单"段列明常用调用
- 新增脚本前先查 `工具/` 是否已有同类（避免重复）
- 一次性脚本（如某次评测）可放 `工具/_oneshot/` 子目录，不混淆主脚本

## 6. 与产品研发型项目的桥接

工具运营型项目常作为产品研发型项目的"客户场景"或"测试床"（如卟卟鸡作为知识自进化引擎的客户场景）：

- **作为客户场景**：在 L2 项目专属内容中标注"本项目作为 XX 引擎的客户场景"，并列明引擎对接接口（MCP / 文件引用）
- **作为测试床**：项目内可保留"被测对象"（如知识库的实体卡片是知识自进化引擎的测试数据）
- **桥接原则**：通过 MCP 协议或相对路径文件引用接入，**不在工具运营型项目内重复实现**产品研发型项目要交付的能力

## 7. 与知识库的双向流动

所有工具运营型项目（包括知识库自身）都参与知识库联动：

- **查询**：研判前必先 qmd 查 `~/Desktop/AILab/知识库/wiki/`（铁律见 L1 §9.2）
- **回写**：达到阶段性结论时按 L1 §6.2 触发知识库刷新
- **联动**：知识库摄入新情报时，若与某工具运营型项目相关，主动提醒主人是否同步

## 6. 反馈日志机制（工具运营型专属）

> 工具运营型项目持续运行、没有五段流程的阶段日志承载错误归档，因此每个项目根目录维护 `反馈日志.md`，沉淀主人的纠正/意见和工具/流程错误，用于事后复盘和持续改进。
>
> 产品研发型项目**不适用**本机制——错误归档到各阶段 `决策日志/` 与 `讨论记录/`。

### 6.1 触发规则

**自动触发**（识别到立即写入）：

| 类型 | 信号 | 必/选 |
|---|---|---|
| 🔴 错误 | Agent 输出与主人意图相反 | 必记 |
| 🟡 纠正 | 主人说"错了/不对/不是这样/应该.../刚才有问题/重新做" | 必记 |
| 🟡 纠正 | 主人重复同一指令 | 必记 |
| 🟢 意见 | 主人说"我觉得.../建议/不如/更好的方式..." | 必记 |
| 🔧 工具异常 | 代码/AKShare/Git 等异常导致返工 | 选记（影响主人体验时） |
| 💡 改进点 | 主人讨论中提到"以后可以.../下次..." | 选记 |

**显式触发**：主人说"记一下/记录/log 一下/留个反馈"

**不触发**：主人正常补充信息、改变需求方向（这是新指令）、Agent 内部自纠、闲聊。

### 6.2 会话结束自检

主人说"再见/收工/告辞/今天就这样/没事了"时，自查：(1) 主人有没有纠正过我？(2) 有没有提过意见或建议？(3) 工具有没有出错且影响体验？有则批量补写，没有则不打扰。

### 6.3 写入格式

`反馈日志.md` 的"反馈记录"小节，每条含：类型、上下文、主人原话、根因、改进方向、状态（待处理/已处理）。

### 6.4 复盘指令

主人说"复盘日志/看看反馈/反馈复盘"时：读日志 → 按状态/类型汇总未处理项 → 输出整改清单 + 优先级 → 按主人确认逐条处理 → 更新状态字段。

### 6.5 与 MEMORY.md 联动

- 反馈日志记**事件**（一次性，可累积，按项目分别维护）
- MEMORY.md 记**长期偏好**（永久，跨会话生效）
- 同一类反馈反复出现（≥ 3 次）→ Agent 主动建议主人："是否升级为 MEMORY.md 偏好？"

### 6.6 同步 Git

`反馈日志.md` 随 sync.sh 一起推送到对应项目的 GitHub。

## 7. 沉淀规范

- 每次决策/操作有结论，写入对应领域目录（如 `评估记录/`、`讨论记录/`）
- 内容：上下文、决策依据、待复盘项、下一步
- 反馈日志见 §6
- 知识库回写见 L1 §6.2
