手机就是工位：Boris 操控上千 Agent 的工作法拆解，与主人专属落地方案

基于 Claude Code 之父 Boris Cherny 在 Sequoia AI Ascent 2026 的访谈、其公开的 howborisusesclaudecode.com，以及业界多 Agent 编排实践，整理出可直接照搬的命令与分阶段方案。

5–10手机端并行会话
数千夜间同时跑的 Agent
100他预测一年后 harness 只剩这么多行

🎯 一句话结论

Boris 不是"手速快"，而是把自己从"写代码的人"升级成"调度 Agent 的人"。靠三样东西放大产能：并行会话 + 子 Agent（横向铺开）、/loop 与 Routines（时间轴上自动循环）、CLAUDE.md + 验证回路（让 Agent 不用盯也能干对）。手机只是入口——真正的杠杆是这套"发起→自动循环→自我验证"的闭环。

01他到底怎么"用手机工作"

手机不是用来打字写代码，而是用来"发起任务 + 巡检状态 + 拍板"。

物理动线（他现场演示的）：

  • 打开 Claude App → 左侧 Code 标签页
  • 里面同时跑 5–10 个会话，每个会话下面再挂一堆子 Agent → 合计几百个在跑
  • 白天用手机发起任务，回到电脑上接力（session 在设备间无缝流转，--teleport）
  • 晚上放几千个 Agent 跑更深的任务，睡一觉醒来收结果

关键不是"屏幕小也能写代码"，而是任务被拆成可异步跑的小单元后，人只需要在"需要输入"时介入——靠系统通知提醒，靠 Agent 视图（claude agents）按 需要输入 / 运行中 / 已完成 三态扫一眼。

三个让手机够用的细节

🎙 语音代替打字

他大部分编程靠"说"。理由：说话比打字快 3 倍，而且口述的 prompt 反而更详细。手机用 iOS 听写，CLI 里按住空格 /voice。

🤖 auto 模式 + focus

手机上不可能频繁点"允许"。他全程 auto 模式（不停下来要权限），配 /focus 只看最终结果、隐藏中间工具调用。安全靠预授权白名单而非 --dangerously-skip-permissions。

🔁 /rewind 而非纠正

Agent 跑偏时，双击 Esc → /rewind 丢掉这次尝试，而不是在聊天里掰扯。坏上下文不进窗口，质量才稳。

📲 手机发起 → 电脑接力

/remote-control 从手机连本机起新会话；& 把任务转后台。碎片时间发任务，整块时间审结果。

02"上千个 Agent"是怎么控住的

不是一个个手动开。靠一座五层金字塔——越往下放大倍数越大，人盯的越少。

L1
单会话 + 子 Agent
一个会话内派子 Agent 做隔离子任务，最近还嵌套到 5 层（agent 开 agent，管理上下文）
×几十
L2
并行会话 + git worktree
claude --worktree 同时开 3–5 个隔离工作树，别名 za/zb/zc 一键跳；claude.ai/code 上再挂 5–10 个
×数百
L3
/loop 定时循环
Claude 用 cron 调度，每分钟/每5分钟/每小时跑一次，最长挂 3 天。同时跑几十个 loop
×全天候
L4
Routines / /schedule 云端
服务器端的 loop，关掉笔记本照跑；可由 cron、GitHub 事件、API webhook 触发
×离线持续
L5
Dynamic Workflows（动态工作流）
说一句"use a workflow"，编排器写脚本，一次会话拆出几百个并行子 Agent（单次最多 1000，同时 16 个），各自在独立 worktree 干、审查 Agent 验、再合并
×上千

让 Agent "不用盯"的三根支柱

1️⃣ CLAUDE.md = 复利工程（Compounding Engineering）
"每次 Claude 犯错，我不在聊天里纠正它，而是让它把规则写进 CLAUDE.md。" 口头纠正只修这一次，写成规则修所有未来。团队每周更新多次，PR 里 @.claude 直接把教训沉淀成规则。

2️⃣ 验证回路 = 质量 2–3 倍
"给 Claude 一个能自我验证的方式，它会把最终质量提高 2–3 倍。" 浏览器插件实测 UI、跑测试套件、/goal 设定"测试全过+lint干净才准停"的完成条件。

3️⃣ 上下文极简主义（Context Minimalism）
"给最小的系统提示、最小的工具集，然后让模型自己想办法。" 告诉它目标，而不是每一步。模型越强，harness 越该缩小。

"我的 Claude 们全天在跑。遇到不确定的问题时，它们会通过 Slack 跟别人的 Claude 对话来解决。"

—— Agent 之间用 Slack MCP 自主通信协作。4.7 模型甚至会自发用 loop：发现数据在变，主动说"我每 30 分钟给你发份报告"，还自己调 Slack MCP 推送。

Boris 真实在跑的 loop（可直接抄）

命令干什么
/loop 5m /babysit看护 PR：自动改 code review 意见、自动 rebase、推进合并

03其他人怎么玩多 Agent

同一套思路，业界已有标杆案例和开源工具。

🦀 Bun：Zig → Rust 整库重写

Jarred Sumner 用 Dynamic Workflows 把 Bun 移植到 Rust：75 万行 Rust、测试通过率 99.8%、11 天。做法 = 拆成单元 → 每单元一个子 Agent 在独立 worktree 干 → 每个文件配 2 个审查 Agent 验证 → 合并。这就是 L5 的极致样板。

🌲 Parallel Worktrees 开源技能

把 git worktree（共享一个 .git、隔离工作目录）+ N 个并行 Agent 组合：跑 N 个就有 N 个有效解，挑最好的。把"LLM 的不确定性"当成特性用。

🧠 pro-workflow / 自纠错记忆

社区方案把"从纠正中学习"做成系统：自纠错记忆跨 50+ 会话复利、上下文工程、Agent 团队、十几个实战技能——本质都是 Boris 那套 CLAUDE.md 复利的工程化。

📜 确定性编排（Deterministic Workflows）

业界共识：把"该并行什么、该验证什么、该合并什么"写进脚本，用确定性的控制流（循环、条件、扇出）管 Agent，而不是让模型自由发挥。这正是 Claude Code 内置 Workflow 的设计——一个脚本调度上千 Agent。

04主人专属落地方案

主人是手机优先的单兵研究者，不是带团队刷 150 PR/天。所以不照搬规模，照搬结构：先把"发起→循环→验证"闭环搭起来，再逐层加并行。

核心取舍：主人多数任务是调研 / 写作 / 前沿技术拆解，不是大型代码库重写。所以重点放在 L3（/loop 定时调研）+ L1（子 Agent 拆研究）+ 三支柱（记忆复利），L5 的千 Agent 留给真正的大工程。先把杠杆用在刀刃上。

第一周 · 现在就做搭好"记忆复利 + 验证"地基

  • 用好已有的 memory 机制：主人这套已有 MEMORY.md 自动记忆——继续坚持"每次纠正→写成规则"，而不是口头改。这就是 Boris 的复利工程，主人已经在做了。
  • 给每类任务定 /goal：调研任务 → "回答完这 5 个问题、每条有出处才算完"；写作 → "读者是谁、交付什么格式才算完"。把模糊任务变成可验证目标（正好契合主人 CLAUDE.md 的"目标驱动"）。
  • 手机三件套：auto 模式 + /focus + 语音口述。碎片时间用语音发起任务。

第二～三周 · 加循环上 2–3 个 /loop 把重复劳动自动化

挑主人真实重复的活，先跑这几个（举例，按需改）：

主人的 loop作用
/loop 1d /tech-radar每天扫指定信源（X / arXiv / 微信公众号）抓前沿动态 → 聚类 → 出一页简报
/loop 12h /inbox-research把主人攒的待研究清单逐条深挖，每条产出带出处的小结
/loop 1d /repo-watch盯几个 GitHub 项目的更新，有大改动就拆解能借鉴什么（呼应主人的 borrow-github-module 技能）

关掉电脑也想跑的，升级成 /schedule（云端 Routine）。结果让它推到主人能扫读的地方（如固定一个汇总文件 / 通知）。

一个月后 · 铺并行L1 子 Agent 拆研究 + L5 留给大工程

  • 调研用扇出：一个大题目 → 派多个子 Agent 各查一个角度（已有 deep-research 技能正是这个），主 Agent 汇总。一次得多个独立视角，再挑最好的。
  • 真遇到大工程才上 Workflow：比如"把某开源模块整体移植/重构进自己项目"——这时说"use a workflow"，让它拆单元、并行子 Agent、配审查 Agent 验证再合并（Bun 那套）。日常调研别杀鸡用牛刀。
  • 多会话巡检：手机上同时挂 3–5 个会话，靠通知 + Agent 视图三态扫一眼，只在"需要输入"时介入。

⚠ 三条避坑（诤臣之言）

  • 别一上来追"上千 Agent"的数字。Boris 有团队 CLAUDE.md、成熟 CI、海量预授权命令托底；主人单兵直接铺千 Agent 只会失控烧钱。先把闭环跑顺，规模自然长。
  • auto 模式 ≠ 裸奔。务必用预授权白名单（/permissions 存到 settings.json）而不是 --dangerously-skip-permissions；重大/不可逆操作仍要人来拍板。
  • 验证回路是质量的命门。没有自我验证的 loop，跑得越多错得越多。每个自动任务都要带一个"怎么算对"的检查。

05命令 & 配置速查卡

手机/电脑通用，按需取用。

并行与会话

# 开隔离工作树并行干活
claude --worktree my_feature
# 命名会话，方便 Agent 视图里扫读
claude --name "tech-radar"
# 给多仓库访问权
claude --add-dir /path/to/other-repo
# 看所有并发会话（需要输入/运行中/已完成）
claude agents
# 手机连本机起新会话
/remote-control

循环与目标

# 定时循环（最长挂 3 天），间隔可 5m / 30m / 1h / 1d
/loop 1d /tech-radar
# 云端 Routine：关电脑照跑，可被 cron / GitHub / webhook 触发
/schedule …
# 设完成条件，不达成不停
/goal 这5个问题都回答完且每条有出处
# 大工程：拆单元→并行子Agent→审查Agent验证→合并
"use a workflow"

质量与上下文

# 跑偏了：丢掉这次尝试（双击 Esc）
/rewind
# 长会话压缩并给方向
/compact 聚焦 auth 重构，丢掉测试调试那段
# 隐藏中间工具调用，只看结果（手机友好）
/focus
# 最难的问题给最大推理预算（仅当前会话）
/effort max

📌 主人三步起手

① 今天：把一类重复调研任务定成 /goal + 试跑一个 /loop 1d。
② 本周：坚持"纠正即写规则"，让 memory 复利起来。
③ 本月：遇到大题目用子 Agent 扇出（deep-research），遇到真·大工程才上 workflow。

规模是结果，不是目标。先把"发起→循环→验证"这个闭环跑顺，主人自然也能从"做事的人"变成"调度 Agent 的人"。
