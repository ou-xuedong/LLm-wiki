# 使用指南 · 画像自进化框架

> 两件事：① 用框架**快速创建**一个新项目；② 框架更新后**升级改造**已有项目。
> 框架原理见 `框架设计.md`，可执行规则见 `模板/CLAUDE.md`。

---

## 一、快速创建新项目（约 5 步）

1. **建目录骨架**
   ```bash
   mkdir -p <项目名>/{贴源层,每次的讨论总结,人物画像,产出文件/backUp}
   ```

2. **拷规则模板**：复制 `模板/CLAUDE.md` → `<项目名>/CLAUDE.md`，把所有 `<项目名>`/`<对象>`/`<主题>` 占位符替换为实际值，删除顶部说明注释。保留 frontmatter 里的 `框架: 画像自进化框架 v1.0.0`（升级时要比对）。

3. **拷画像骨架**：复制 `模板/人物画像_模板.md` → `<项目名>/人物画像/<对象>.md`，建空文件 `<对象>_更新日志.md`（首行写索引格式说明）。

4. **第一次讨论即走三件套**：① 贴源归档 → ② 写讨论总结 → ③ 刷新画像 + 追加索引。（图片记得转文字后删原图。）

5. **验证**：新开一次会话，确认 AI 只读「CLAUDE.md + 画像 + 最近 1 次总结」就能接上线、且不全量翻历史。✅ 通过即框架生效。

> 多对象：同一项目可并存多份 `<对象>.md` + `<对象>_更新日志.md`，共用四层结构与加载策略。

---

## 二、升级改造已有项目（框架版本更新后）

**目的**：框架设计迭代后（如 v1.0.0 → v1.1.0），把旧项目平滑升级到最新设计，**不丢历史、不破坏已有画像**。

### 升级流程
1. **读 CHANGELOG**：看从旧项目当前版本到最新版**改了哪些点**（CHANGELOG 每条标注"是否需迁移动作"）。
2. **认旧版本**：看旧项目 `CLAUDE.md` 顶部 frontmatter 的 `框架:` 版本号；无版本号者按 v1.0.0 前的"古早版"处理。
3. **先备份**：升级前把旧 `CLAUDE.md` 存为 `CLAUDE.md.bak`、画像存为 `.bak`。
4. **分类改动**，逐项落地（见下表三类）。
5. **更新版本号**：改完把 `CLAUDE.md` frontmatter 的 `框架:` 行升到最新版。
6. **验证**：跑一次"加载策略"验收——新会话只读必读层能否接上线。

### 改动分三类处理
- **纯规则类**（落盘/命名/加载规则变化）→ 直接用新模板覆盖 `CLAUDE.md` 的对应段落，**项目状态内容不动**（状态只在画像里，规则文件本就不含状态，覆盖安全）。
- **结构类**（目录/文件拆分变化，如"更新日志拆出索引层"）→ 按新结构**迁移已有内容**，迁移后在更新日志记一行"治理动作"。这类**必须人工核对**，CHANGELOG 会标 ⚠️ 需迁移。
- **画像内部约定类**（新增标记体系、新增顶部块如"阅读前提/决策矩阵"）→ **增量补**到现有画像，不重写历史结论。

### 升级铁律
- 🔴 **状态内容只在画像**：升级动的是"规则与结构"，绝不借升级之名改写"对该对象的结论"。
- 🔴 **先备份再改**，结构类迁移后必须核对"四层不重复"仍成立（同一结论没被迁成两份）。
- ⚪ 旧项目若已偏离框架很多，优先**保住画像正文**（长期记忆），规则/结构按新版重建。

## 三、健康自检（任何时候可跑）
- [ ] 同一结论是否只在一层权威存放？（查②③有无复述）
- [ ] 画像顶部"最近更新"是否最新在前、每次都追加？
- [ ] 更新日志是否一行一条、不复述内容？
- [ ] 新会话固定开销是否只 = 规则 + 画像 + 最近 1 次总结？
- [ ] 贴源层是否还留着未转文字的图片？（应转完即删）
