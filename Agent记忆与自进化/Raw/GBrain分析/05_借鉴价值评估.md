### 1.1 🟢 takes vs facts 二分模型 — 认知模型创新

**算法层（What）**

把记忆分成两张表，靠 `holder` 字段区分"谁的认知"：

```python
# 主人简化版（v0.1）
facts: 主人本人对话即时抽取的事实
  fields: id, fact, kind, source_session, confidence, embedding, created_at, expired_at
  kind: preference | commitment | event | belief
  holder: 隐含 = brain_owner（即主人自己）

takes: 主人引用的外部观点 / 第三方信念
  fields: id, claim, holder, kind, weight, embedding, source, active
  holder: 'self' | 'expert:xxx' | 'paper:yyy' | ...
  kind: claim | observation

# 桥接（v0.2+）：凌晨 consolidate 阶段把同主题 facts 簇合并升级为 take
facts.consolidated_into → takes.id
facts.consolidated_at → datetime
```

**工程胶水层（Why It Works）**

- **不分会出三类灾难**：归因塌陷（别人观点算成主人的）/ 置信度错配 / 检索污染
- **kind 起步只要 2 种**（gbrain 用 9 种是 100K 页演化出来的），主人 v0.1 用 2 种够
- **holder 字段是不能省的核心创新**——多视角支持的根
- **takes 内嵌 1536d embedding + HNSW（active only）**——支持 take 间语义检索（"主人说过的关于 X 的看法 vs 业界专家说过的"）
- **生产实测成本**（gbrain ~100K 页 brain）：$361 一次全量抽取（Azure GPT-5.5 = Opus 1/8 价）

**框架耦合层（Glue）**

- 依赖 Postgres / SQLite + pgvector（或 sqlite-vss）
- 依赖一个 Sonnet 级模型做 fact extraction（每轮对话调一次）
- 不依赖 gbrain 任何源码 — 纯设计模式可平移

**借鉴姿势**：① 抄思想 → 在 Python 项目里**重新实现 schema**

**契合度评分**：🟢 几乎 100% 契合主人新方向

**踩坑预警 ⚠️**：
- **Holder ≠ Subject**：「Garry has a hero/rescuer pattern」的 holder 应该是 `brain`（观察者）而不是 `people/garry`（被观察对象）——gbrain 首次跑出 6.5/10 attribution 分。Prompt 要显式举反例。

参见 `03_数据与交互视角.md` C 节（C.1-C.5）+ G.3

### 1.2 🟢 自连线（Self-wiring）— 90% 零 LLM 的图谱构建

**算法层（What）**

每次 `put_page` 写入时，确定性地从正文 + frontmatter 抽出"实体引用 + 类型化边"，写入 `links` 表。**零 LLM 调用**。

```python
def auto_link_on_write(page):
    # 路径 1：markdown 正文
    candidates = WIKILINK_RE.findall(page.body)         # [[people/sam-altman]]
    candidates += ENTITY_REF_RE.findall(page.body)      # [Sam Altman](people/sam-altman)

    # 路径 2：frontmatter map（11 条静态规则）
    for field, link_type, direction in FRONTMATTER_LINK_MAP:
        if field in page.frontmatter:
            candidates += build_edge(field, page, link_type, direction)

    # link_type 推断：4 条 verb 正则在 240 字符上下文里跑
    for c in candidates:
        c.link_type = infer_link_type(context_window(c, 240))
        # FOUNDED_RE > INVESTED_RE > ADVISES_RE > WORKS_AT_RE > mentions

    write_links_table(candidates)  # UPSERT
```

**工程胶水层（Why It Works）**

- **类型化边精度 70-94%**——靠 4 条 verb 正则 + 3 条 role-prior 正则（不靠 LLM）
- **frontmatter `incoming` direction 设计很妙**：`key_people: [Pedro]` 写在 `companies/stripe.md`，自动生成 `people/pedro → stripe (works_at)`，**主谓宾正向语义**
- **幂等可重算**：`extract --rebuild` 删掉所有 frontmatter-derived 边 + 重跑全部 .md，整张图重建；单页 put_page 时精准只清自己产出的边
- **pg_trgm 模糊匹配 + slugify fallback**——显示名 → slug 的零 LLM 解析
- **UNIQUE NULLS NOT DISTINCT 约束**——同一对节点的 markdown 边和 frontmatter 边并存而不冲突

**框架耦合层（Glue）**

- 依赖 Postgres `pg_trgm` 扩展（SQLite 用 `fuzzywuzzy` 库可替代）
- 依赖一份**静态正则字典**（FRONTMATTER_LINK_MAP 11 条 + verb 正则 4 条）
- 不依赖任何 LLM
- 不依赖 gbrain 源码，**完全可移植到 Python**

**借鉴姿势**：① 抄思想 + 重写正则字典

**契合度评分**：🟢 主人若不抄，知识库会退化成孤儿页堆栈

**主人裁剪建议**：
- verb 正则起步 1-2 类够（统一 `mentioned`，等真需要区分 invested_in / advises 再加）
- frontmatter map 起步 5 条（people / company / tag / source / parent）

参见 `03_数据与交互视角.md` D 节（D.1-D.4）+ G.2

### 1.3 🟢 Hybrid Ranking 公式 — 召回融合 + 三轴增权

**算法层（What）**

gbrain 的 query 引擎核心公式（**直接拿走，5 行 Python 就能写**）：

```python
# 第一步：RRF 融合（FTS + 向量双源）
RRF_score(item) = Σ_lists 1 / (60 + rank_in_list)  # K=60 默认

# 第二步：归一化 + compiled_truth boost
boosted_rrf = normalize(RRF_score) × (2.0 if source == 'compiled_truth' else 1.0)

# 第三步：cosine 重排（向量真跑了的话）
blended = 0.7 × normalize(boosted_rrf) + 0.3 × cosine(query_emb, chunk_emb)

# 第四步：post-fusion 三轴增权（multiplicative）
final = blended
  × (1 + 0.05 × ln(1 + backlink_count))     # 反向链接增权
  × (1 + 0.15 × ln(1 + salience_score))     # salience 增权
  × (1 + recency_decay(slug_prefix, age))   # recency 衰减（按 slug 前缀配置 halflife）

# salience 简化版：salience_score = emotional_weight × 5 + ln(1 + take_count)
```

**工程胶水层（Why It Works）**

- **三轴都用对数压缩**（log）——避免"超级节点拉爆整个 ranking"，同时保留"重要节点排名上浮"的直觉
- **recency 按 slug 前缀**：`media/articles/` halflife=30d、`concepts/` halflife=∞——同一公式覆盖"时效新闻"和"长期信念"两种语料
- **K=60 是 RRF 论文经验值**——不要拍脑袋改
- **0.7 / 0.3 权重**：RRF 主、cosine 副。cosine 拉太高会让"语义相关但内容浅"的页超过"关键词精确命中"的页
- **gbrain 实测 P@5 49.1% / R@5 97.9%**：召回近天花板，精度仍差——**说明缺重排器**（Cohere / Voyage rerank-2 是 v0.2 的下一步）

**框架耦合层（Glue）**

- 依赖一个 FTS 引擎（Postgres FTS5 / SQLite FTS5）
- 依赖一个向量索引（pgvector / sqlite-vss / chromadb）
- 依赖反向链接 count（来自 links 表）
- **完全可在 Python 重写**，gbrain 的 `hybrid.ts` 总共 ~50 行核心逻辑

**借鉴姿势**：① 抄思想 + 公式直译

**契合度评分**：🟢 简单到无理由不抄

**主人裁剪建议**：
- v0.1：先做 RRF + 反向链接增权两个（5 行代码）
- v0.2：加 salience（emotional_weight 简化为"主人显式打星 + take 引用数"）
- v0.3：加 recency 衰减 + 上 Cohere reranker

参见 `03_数据与交互视角.md` E 节（E.1-E.3）+ G.5

### 1.4 🟢 Engine 双轨抽象 — 零运维本地 + 可选云

**算法层（What）**

定义一个统一的 `BrainEngine` 抽象，Postgres / PGLite 双实现共存：

```typescript
// gbrain 原版（TS）
interface BrainEngine {
  searchKeyword(query, options): Promise<SearchResult[]>
  searchVector(emb, options): Promise<SearchResult[]>
  putPage(slug, body, fm): Promise<Page>
  getPage(slug): Promise<Page | null>
  // ... 47 个 op
}
```

```python
# 主人 Python 版（简化）
class BrainEngine(Protocol):
    async def search_keyword(self, q: str, **opts) -> list[SearchResult]: ...
    async def search_vector(self, emb: list[float], **opts) -> list[SearchResult]: ...
    async def put_page(self, slug: str, body: str, fm: dict) -> Page: ...
    async def get_page(self, slug: str) -> Page | None: ...
    # ...

class LocalEngine(BrainEngine): ...   # SQLite + sqlite-vss
class CloudEngine(BrainEngine): ...   # Supabase Postgres + pgvector
```

**工程胶水层（Why It Works）**

- **gbrain 的关键魔法**：PGLite 不是 SQLite 仿 Postgres，是 **WASM 编译的真 Postgres 17.5**（@electric-sql/pglite 0.4.3）——HNSW / tsvector / pg_trgm / advisory lock / 递归 CTE 全部共享。让 `gbrain migrate --to supabase` 双向无损可行
- **Python 没有等价物**——最接近是 DuckDB，但**不是 PG 方言**。主人若用 Python：本地 SQLite + sqlite-vss，云 Supabase/Neon，**schema 一致性靠对齐 + 一行迁移脚本**保证
- **markdown 仓做 system of record**：DB 出问题 `rebuild` 一键重建（gbrain 在 `docs/architecture/system-of-record.md` 把每张表强制归为 FS-canonical / Derived-from-FS / DB-only 三类之一，CI 卡门）
- **零运维本地是默认**：新用户 2 秒就绪（PGLite 起进程内 DB），重度用户一行命令迁云

**框架耦合层（Glue）**

- 主人 Python 版替代品候选：
  - 本地：SQLite + `sqlite-vss` / `chromadb` 嵌入式 / DuckDB
  - 云：Supabase / Neon / Postgres + pgvector
- 不依赖 gbrain 源码

**借鉴姿势**：① 抄思想 + Python 重新设计接口

**契合度评分**：🟢 主人项目最该照搬的**产品策略**（不只是技术）

**主人架构建议**：
```
~/.agent-memory/
├── facts/              # db_tracked（git）— 主人偏好
├── notes/              # db_tracked — 长记忆笔记
├── raw/                # db_only — 原始对话日志（机器产出，不进 git）
└── .agent-memory.db    # SQLite + sqlite-vss 索引（DB 是 derived cache）
```

**铁律建议**：主人项目从 day 1 立"DB 可破坏可重建"契约——`agent-memory rebuild --confirm-destructive` 命令必须能从 fs 完全重建 DB。备份/迁移/容灾全部退化为 `git pull`。

参见 `04_技术落地与运维视角.md` C 节 + H.1

### 1.5 🟢 评测三件套 — BrainBench-Real 优先

**算法层（What）**

三层评测体系，按"成本递增、价值递减"排序：

| 评测 | 数据来源 | 频次 | 目的 |
|---|---|---|---|
| **BrainBench-Real** | 用户真实查询（`eval_candidates` 表，PII 脱敏） | 每次 PR | 防回归（自己代码改动有没有破坏 retrieval） |
| **LongMemEval 内置** | 公开 HuggingFace 数据集 | 每周 / CI | 横向可比（业界基准） |
| **BrainBench fixture** | 240 页 Opus 生成的 rich-prose 语料 | 版本发布 | 纵向追踪（P@5/R@5 数字） |

**工程胶水层（Why It Works）**

- **capture 默认 OFF**——`GBRAIN_CONTRIBUTOR_MODE=1` 才开，避免普通用户被惊吓
- **LongMemEval 用 in-memory PGLite + TRUNCATE 重置**：每题独立环境，schema 迁移安全
- **CI 卡门 5 行 bash**：`jq -e '.summary.mean_jaccard >= 0.85'` 走 `gbrain eval replay`
- **三个指标 hand-off**：Jaccard@k（捕获 vs 当前 slugs）+ top-1 stability + latency Δ
- **prompt-injection 防御一源**：`INJECTION_PATTERNS` 在评测和生产共用

**框架耦合层（Glue）**

- 依赖一个 `eval_candidates` 表（schema 简单）
- 依赖 LongMemEval 数据集（HuggingFace 拉取）
- 完全可平移到 Python — 用 pytest + jsonl 就能复刻

**借鉴姿势**：① 抄思想

**契合度评分**：🟢 强烈推荐，从 V0 就内建评估骨架，**不要等产品成熟才补**

**主人裁剪建议**：

1. v0.1 直接加 BrainBench-Real（最低 ROI 最高） — capture 几行 SQL，replay 几十行 Python
2. v0.2 内嵌 LongMemEval（已有公开数据集，零标注成本）
3. v0.3+ 自造 BrainBench fixture（需要人工标 qrels，前期不必）

参见 `04_技术落地与运维视角.md` F 节 + H.2

## 2. 🟢 应当借鉴的工程模式

### 2.1 dispatch.ts 契约优先 — 四 transport 收口

```typescript
// gbrain 原版：dispatch.ts 仅 279 行
function dispatchToolCall(
  engine: BrainEngine,
  name: string,
  params: any,
  opts: { remote, auth, takesHoldersAllowList, sourceId, metaHook }
): Promise<any>
```

**意义**：stdio MCP / HTTP+Bearer / HTTP+OAuth / CLI **四种 transport** 全部走这一个函数，五字段 opts 把"我是谁、我从哪来、我能看什么"统一抽象。**契约优先架构的范本**。

主人 Python 版做法：
```python
async def dispatch(
    engine: BrainEngine,
    op_name: str,
    params: dict,
    ctx: OperationContext,  # remote, auth, sources_allow, meta_hook
) -> dict:
    op = OPS_REGISTRY[op_name]
    return await op(engine, params, ctx)
```

**借鉴姿势**：① 抄思想 — Python 用 Pydantic + ops 字典 50 行复刻

参见 `04_技术落地与运维视角.md` I.2

### 2.2 Trust Boundary — `remote` / `auth` 编译期固化

`OperationContext.remote` 字段把"本地 CLI 信任 vs Agent 远程不可信"做成**编译期可校验语义**：
- 敏感操作如 `file_upload` / `submit_job` 在 `remote=true` 时默认 closed
- failure 模式 = closed by default

主人项目从 day 1 必须有这个字段。**事后补极痛**。

参见 `01_业务与用户视角.md` C.2

### 2.3 Markdown-as-truth + db_tracked / db_only 分层

- **核心知识（人写）→ db_tracked**：进 git，可读可备份
- **机器生成（转录/推文/原始日志）→ db_only**：不进 git，避免污染
- **DB 是 derived cache**：丢失可从 fs `rebuild` 重建

主人版结构：
```
~/.agent-memory/{facts,notes}/    ← db_tracked，进 git
~/.agent-memory/raw/              ← db_only，git ignore
~/.agent-memory/.db               ← derived cache
```

参见 `03_数据与交互视角.md` G.6

### 2.4 Thin Harness, Fat Skills — 引擎不解释，skill 是 fat markdown

- **核心层（retrieval + ranking + storage）零 LLM call**——只确定性逻辑
- **LLM 在外层 Agent / Skills 里调**，进核心层只通过结构化接口
- **Skills 是 markdown 文件**，Agent 直接 read，**不要把 prompt 编译进二进制**

⚠️ **诤臣提醒**：Garry 押注的是"Opus 级模型能从英文规范实现复杂能力"——主人若面向较弱模型或纯中文场景，**关键路径要补确定性 fallback**（Skill 作为高层 DSL，关键路径有 code 兜底）。

### 2.6 反向链接铁律（Iron Law）

**Rule**：每提到一个有页面的实体，对方页面**必须**回链 → 写入 timeline section

**落地三件事**：
1. 每条事实必带 `[Source: ...]`
2. `put_page` 钩子自动抽实体（不能依赖手工 `add_link`）
3. back-link append 到对方页面的 Timeline section

主人 AILab 规范 §3 的"知识库回写"流程不接这条，长期一定退化成孤儿页堆栈。

## 3. 🟢 应当借鉴的产品机制

### 3.1 Tier 1/2/3 升级 — 提及计数自动驱动

```
实体首次提及 → notability gate → Tier 3 stub（仅 cross-ref）
  ↓ 多次互动累积
Tier 2 notable（web research + social）
  ↓ 核心圈/合作者升级
Tier 1 key（全管线 + 深度研究）
```

**关键设计**：
- 不上来就全量丰富 — 低价值实体仅留 stub，避免 API 浪费
- 频次自动升级 — 不需要主人手动标"重要"
- 跨多次互动后系统识别为关键人物 → 升级到 Tier 1 全量画像

主人项目几乎所有"实体丰富"流程都应当照这条做（公司、人物、概念、产品）。

参见 `02_功能与流程视角.md` B.2 + E.2

### 3.2 Dream Cycle 工程化降本 — 两层模型 + 沙箱

**完整工程化设计**（不只是"凌晨跑维护"）：

| 设计点 | 实现 | 价值 |
|---|---|---|
| **两层模型** | Haiku 廉价过滤"值不值深度处理" → Sonnet 深度合成 | 成本压到 ~$1/day |
| **冷却** | `cooldown_hours: 12` | autopilot 每天最多 2 次 |
| **幂等键** | `(file_path, content_hash)` | 编辑后转录生成新 slug 不覆盖 |
| **沙箱写入** | `allowed_slug_prefixes` allow-list | 即使 prompt injection 也写不出指定目录 |
| **隐私正则** | `exclude_patterns: ["medical", "therapy"]` | 在 LLM 调用**之前**过滤 |
| **Patterns 阶段** | 跨 ≥3 reflections 找纵向主题 | 长期记忆产出物 |

**主人借鉴度** 🟢🟢🟢——这是 GBrain 整套设计里**最具差异化、最值得抄**的部分。Letta/MemGPT/Mem0 都没做到这个工程深度。

⚠️ **主人若不抄**：跑大规模长期记忆时**一定**会撞墙（成本爆炸 / prompt injection / 数据污染 / 重复消费 loop）

参见 `02_功能与流程视角.md` B.3 + E.5 + `03_数据与交互视角.md` F 节

---

### 3.3 ⭐ Skillify（procedural memory）— 主人新方向的**关键差异化**

**这是 GBrain 最被低估、对主人最稀缺的差异化**。

**市面对比**：
- 主流 Agent 记忆系统（Letta / Mem0 / Zep）= 只做 **declarative memory**（记事实）
- GBrain 用 skillify 做 **procedural memory**（记"怎么做"）

**Skillify Loop**：
```
Agent 执行失败 / 用户喊 "skillify it!"
  → Audit 11 项检查清单
  → 写 SKILL.md + 提取确定性脚本
  → Cross-modal eval（3 个 provider 前沿模型 5 维评分）
  → Cycle 1-3 应用 top-10 改进
  → 通过后写测试锁定好行为
  → 入 RESOLVER.md 注册触发短语
  → routing-eval 验证下次同样问题自动命中
```

**主人裁剪建议**：
- 起步 **3 项必备**（不要 11 项）：SKILL.md + 触发短语进 RESOLVER + 一个 e2e 测试
- Cross-modal 三模型 eval **不抄**（成本高，前期 Anthropic 单家就够）
- **核心精神保留**：一次失败 = 一次系统缺陷 = 一个新 skill 的机会；**测试在 eval 之后写**，避免锁住 mediocrity

⚠️ **奴婢诤臣一句**：主人项目题目是"Agent 记忆与**自进化**"——"自进化"的最具体落地就是 **procedural memory**。**不抓住这个就跟所有 RAG-only 项目同质化**。

参见 `02_功能与流程视角.md` B.4 + E.3

```yaml
allowed_slug_prefixes: ["originals/", "patterns/", ...]    # 白名单写入
```

主人若做 personal agent，**这两条必须有**。

## 4. 🟡 谨慎借鉴 / 简化引入

### 4.1 Dream Cycle 12 phase — v0.1 不做，v0.3 起 consolidate + embed

gbrain 12 phase 太重，主人初期不需要。**最低实现**：

```python
# v0.3 凌晨任务（仅 2 phase）
async def nightly_consolidate():
    # 1. consolidate-only — 聚类昨天的 facts
    clusters = embed_cluster(yesterday_facts, threshold=0.85)
    for c in clusters:
        take = await sonnet_synthesize(c)
        bridge(facts=c, take=take)  # facts.consolidated_into = take.id

    # 2. embed --stale — 给新增 facts/takes 补 embedding
    await batch_embed_missing()
```

**不要碰**（早期）：synthesize / patterns / extract_facts — 都要 LLM subagent + Minions

但 phase 顺序设计（lint → sync → extract → consolidate → embed → purge）是教科书级**依赖关系驱动**——主人扩展时按这个顺序加。

参见 `03_数据与交互视角.md` G.4

---

### 4.2 50 skills 全量 — 起步 10-15 个，分 4-5 层

**最小可工作核心**：

| 层 | 必备 skill | 类比 gbrain |
|---|---|---|
| 信号层 | 1 个 always-on 捕获 | signal-detector |
| 读循环 | 1 个 brain-first 查询 | brain-ops + query 合并 |
| 写循环 | 1 个 ingest（含路由） | ingest + enrich 合并 |
| 维护 | 1 个 dream cycle 简版 | maintain |
| 元能力 | 1 个 skillify 简版 | skillify |

剩下按主人实际用法（投资决策？知识库？autoResearch？）按需长出，**不要预先全建好**。

参见 `02_功能与流程视角.md` E.1

### 4.3 always-on signal-detector — 按场景

- ✅ 本地对话 / IDE 工作流：always-on 合理（不阻塞主响应，平行 spawn）
- ⚠️ 手机端 / 移动场景：成本敏感，改"高频但有冷却"（5 分钟内同实体不重复触发）
- ❌ 批量 import 场景：必须关掉，走批处理路径

**隐性铁律**：always-on **必须**是廉价模型 + 不阻塞主响应。主人若用 Opus 跑 always-on 一天烧 100 刀。

参见 `02_功能与流程视角.md` E.4

## 5. 🔴 不建议抄 / 主人场景不适用

### 5.1 Minions（durable job queue 自建）

- gbrain 4375 行代码 / 9 状态机 / PID 锁 / supervisor 子进程 / advisory lock —— **对个人项目过度工程**
- 主人路线：
  - 第一阶段：同步 RAG，无队列
  - 第二阶段：上 **Celery + Redis** 或 **APScheduler + SQLite-backend**（Python 原生、文档全）
  - 真要 Postgres-native：用 **pgmq** 或 **pg-boss**，不自建
- **唯一值得偷的设计点**：subagent handler 的 **crash-resumable LLM loop**——`messages` 和 `tool_executions` 持久化到 DB，进程崩了从已 commit 轮次继续。这个理念用任何队列都该实现

参见 `04_技术落地与运维视角.md` H.5

## 8. 总结：主人新方向的"GBrain 借鉴 DNA"

经过 4 视角通盘剖析 + 5 类 19 项评估，奴婢给主人的**核心借鉴 DNA**是：

### 8.1 三条不可妥协的设计原则

1. **DB 是 derived cache，markdown 是 source of truth**——day 1 立这条契约，未来省下无数运维痛苦
2. **核心层零 LLM，LLM 在外层**——retrieval / ranking / storage 全确定性，LLM 只在 Skills 和 Agent 顶层调
3. **复利闭环**：每次响应都走 brain-first lookup → write-back → auto-link 三步，**否则不是记忆系统是缓存**

### 8.2 主人项目的关键差异化锚点

**Procedural memory（skillify）**——市面 90% Agent 记忆系统只做 declarative memory（记事实）。主人若把"自进化"做成 procedural memory（记"怎么做"），就跳出 RAG 红海。

### 8.3 主人需要立刻决策的 3 个产品问题

🔴 **必须现在想清楚**（v0.1 启动前）：

1. **商业模式**：主人不是 YC 总裁，纯开源活不下来。先想清楚未来变现路径
2. **技术栈**：Python 还是 TS？（强烈建议 Python，但要面对 PGLite 没有 Python 等价物的现实）
3. **范围边界**：主人新方向 vs `知识存储与自进化引擎` vs `知识库` 三个项目的关系——是合并、是上下游、还是各自独立？
