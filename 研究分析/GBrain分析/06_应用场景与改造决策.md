## 2. 💡 GBrain 的本质 — 不是 PKM，是 Agent 的外脑

读完所有资料，奴婢悟到一句关键定位：

> **GBrain ≠ Obsidian / Notion / Roam 升级版**
>
> **GBrain = 给 AI Agent 用的外脑 + 让 Agent 自己维护 wiki 的引擎**

### 2.1 brain vs memory 二元论（官方原话）

gbrain 官方明确分三层（参见 `docs/guides/brain-vs-memory.md`）：

| 层级 | 性质 | 存储 |
|---|---|---|
| **Brain（世界知识）** | 外部实体事实 | gbrain |
| **Memory（操作状态）** | Agent 自身配置 / 偏好 | Agent Memory |
| **Session** | 当前对话 | 会话窗口 |

✅ **入 Brain**：人物事实 / 公司信息 / 用户对某框架的思考 / 会议记录 / 概念定义
❌ **不入 Brain**：用户偏好简洁格式 / 部署流程配置 / Agent 行为设置

**关键陷阱**：「Pedro 偏好邮件胜于 Slack」看似偏好，但这是关于 Pedro 的**世界知识**，应入 Brain。「用户偏好简洁格式」属于 Agent 如何运作，不入 Brain。

### 2.2 与传统 PKM 的本质区别

| 维度 | 传统 PKM（Obsidian/Notion/Roam） | GBrain |
|---|---|---|
| **维护模式** | 被动存储，主人手动 | 主动学习，Agent 自动 |
| **读写循环** | 写 → 读 单向 | "read before every response, write after every conversation" |
| **结构化** | 平面笔记 | 编译真相 + 时间线 + 证据链 |
| **自进化** | 无（除非主人手动整理） | Dream cycle 自动 |
| **检索** | 关键词 / 反向链接 | Hybrid（FTS + 向量 + 图 + 增权） |
| **更新源** | 主人输入 | 主动整合会议、邮件、日历、推文 |

### 2.3 关键洞察

> "Knowledge management has failed for 30 years because maintenance falls on humans. LLM agents change the equation — they don't get bored, don't forget to update cross-references, and can touch 50 files in one pass."

**主人知识库当前痛点 = 这句话描述的"30 年通病"** —— 维护负担在主人身上，每次新建/修改要套模板、加 [[ ]]、改 index.md、加 log.md、跑 qmd embed。**Agent 出现后，这个痛点终于有解了。**

## 4. 🔍 主人知识库的真实痛点诊断

不是泛泛而谈，是基于主人 `知识库/CLAUDE.md` 规范的**真实痛点**：

### 痛点 1：手动维护负担重 🔴

主人 CLAUDE.md §3.1 的写入流程：
- 消歧搜索 → 套模板 → 改 index.md → 跑 qmd embed → 写 log.md → 关联性扫描 → 加 [[ ]]

**每次新增/修改一个实体，主人要做 7 步**。这是 PKM 30 年通病——维护负担落在人身上。

### 痛点 2：自连线靠"主人/Agent 自觉" 🔴

CLAUDE.md §2.2 弱关联规则：「正文中遇到核心概念**必须**用 `[[ ]]` 包裹」——但这条规则**靠主人/Agent 自觉**遵守。漏一处，知识库就少一条边。

gbrain 的 `put_page` 自动钩子是**强制 + 零 LLM 的**，主人差的就是这一层。

### 痛点 3：检索质量天花板 🟡

qmd 是 FTS + embedding 的基础融合，**没有反向链接增权 / salience 增权 / recency 衰减**。主人的"老牌实体"（被引用 50 次）和"边角实体"（被引用 1 次）在检索结果里**没有差异**——这是检索质量天花板。

### 痛点 4：没有"主动巩固" 🔴

CLAUDE.md §4.2 折旧机制是「**被动减负**」——超期不更新就警告。**没有"主动巩固"**——跨多次讨论涌现的新模式 / 散落在各实体里的相同观点 / 多个概念间的潜在联系——**没有任何机制把它们提炼成新洞察**。

主人 `insights/` 目录里 50 篇综合分析报告，**全是主人手动写的**，没有 Agent 自动合成的。

### 痛点 5：实体一视同仁 🟡

所有实体卡片用同一模板、同一精度。但**热门实体（如 OpenAI、Sam Altman）应该深度画像**，**冷门实体（如某次提到的小公司）只需 stub**——gbrain 的 Tier 1/2/3 自动升级正是这个思路。

主人现在所有实体都按"完整模板"建——这是**对人力的浪费**。

## 5. 🛤️ 三条改造路径

### 路径 A：完全迁移到 gbrain 🔴 不推荐

**做法**：
- 把 `知识库/wiki/` 320 个实体全部 `gbrain import`
- 卸载 qmd，换 gbrain MCP
- 学 gbrain 50 个 skills 体系
- 跑 gbrain dream cycle

**成本估算**：
- 🔴 学习成本：~~小~~ 大（50 skills + Bun 生态 + TS）
- 🔴 迁移风险：320 实体迁移可能丢/错命名规范
- 🔴 技术栈错位：主人系 Python，gbrain 系 TS/Bun
- 🔴 配套生态：要装 OpenClaw 或 Hermes 才能用满，主人没用
- 🟡 运维成本：cron + supabase + 多 API key
- 🔴 商业逻辑：主人项目 `Agent记忆与自进化` **就是要做这个**，迁过去等于放弃做自己的方向

**奴婢判断**：🔴 不推荐。

主人项目目标是**借鉴打底自己设计**，不是**用现成 gbrain**。完全迁移= 放弃自己方向。

---

### 路径 B：保留 qmd 框架 + 借鉴 gbrain 4 个机制 🟢 强烈推荐

**做法**：
保持现有知识库框架（实体模板、frontmatter、qmd 检索、跨项目联动），**只补主人最大的 4 个痛点**：

#### B.1 加自动 [[ ]] 提取钩子（解决痛点 2）⭐

```bash
# 主人现有：知识库写入需要主人/Agent 自觉加 [[ ]]
# 改造后：写入触发 Agent 自动扫描正文 → 提取实体引用 → 写入 relations 字段
```

- **实现路径**：写一个 `qmd autolink` 子命令，调一次主人现有的 qmd embed pipeline
- **价值**：写入即自动连线，主人零负担
- **借鉴 gbrain 的**：`brain-writer.ts` 的 wikilink + frontmatter map 双路径
- **工作量**：1-2 天 Python

#### B.2 加反向链接增权检索（解决痛点 3）⭐

```python
# 主人现有：qmd 检索返回相关度分数（FTS + embedding 融合）
# 改造后：score × (1 + 0.05 × ln(1 + backlink_count))
```

- **实现路径**：用 grep 统计每个实体被多少其他实体 [[ ]]，作为 boost 因子加到 qmd 后处理层
- **价值**：「重要实体」自动排名上浮，检索质量明显提升
- **借鉴 gbrain 的**：`hybrid.ts` 的 5 行公式
- **工作量**：半天 Python

#### B.3 加凌晨"主动巩固"循环（解决痛点 4）⭐⭐⭐

```bash
# 主人现有：月度折旧巡检（被动减负）
# 改造后：每周日凌晨跑 consolidate + patterns（主动巩固）
```

- **实现路径**：
  - 每周扫一次 `知识库/log.md` 看本周新增/修改实体
  - 用 Sonnet 跑两个 phase：
    - `consolidate`：相似实体卡片合并提醒（embedding 相似度 > 0.85）
    - `patterns`：跨多个 insights / log entries 找重复主题，产出新 insight 草稿
  - 写到 `知识库/insights/auto-drafts/`（主人审稿后转正式）
- **价值**：知识库**自己变聪明**——主人睡觉时 Agent 在合成
- **借鉴 gbrain 的**：dream cycle 简版（仅 2 phase，不上 12 phase）
- **工作量**：3-5 天，含 prompt 调优

#### B.4 加 Tier 标记（解决痛点 5）🟡

```yaml
# 主人现有：所有实体一视同仁
# 改造后：frontmatter 加 tier: 1|2|3，按 access_count + 引用数自动升级
```

- **实现路径**：
  - 每月折旧巡检时同步更新 tier
  - Tier 3：仅 stub（最少字段，主人冷启动用模板写两句话即可）
  - Tier 2：完整模板（主人现在的模板）
  - Tier 1：深度画像（额外加 timeline + 关键观点引用）
- **价值**：人力聚焦核心实体，冷实体维护成本降到最低
- **借鉴 gbrain 的**：enrich skill 的 Tier 设计
- **工作量**：1 天 frontmatter 改造 + 主人持续判断

#### B.5（可选）加 takes 表（多 holder 信念）🟢

主人 `insights/` 里包含大量"主人引用的别人观点"——比如 "PG 说创业的本质是..."。这些跟主人本人的判断**应该分开**。

- **实现路径**：
  - 现有 insights 不动
  - 新增 `wiki/takes/` 目录，每个 take 一个小卡片
  - 字段：claim / holder（PG / Sam Altman / 主人 / ...）/ source / weight
- **价值**：跨主体观点对比、引用追溯
- **借鉴 gbrain 的**：takes vs facts 二分（简化版）
- **工作量**：2-3 天 + 持续积累

**总工作量**：B.1-B.4 必做约 **6-9 天**，B.5 可选 **2-3 天**

---

### 路径 C：完全不改 🟡 可接受但错过红利

**做法**：保持现状，继续 qmd + 手动维护。

**理由**：
- 主人 320 实体规模不算大，手动维护尚可
- 主人对自己知识库已经习惯
- 改造有风险

**代价**：
- 维护负担持续在主人身上
- 检索质量天花板低
- 主人**自己在做"Agent 记忆与自进化"项目**——自己的知识库都不进化，跑 RAG 都跑不顺，主人项目说服力会降
