# Agent 驱动研发框架 · 方案设计

> v0.1 草案 · 2026-05-14 · 主人与奴婢小昭共同议定

## 1. 项目定位

- **性质**：基础规则框架（与 `_shared/产品研发型.md`、`_playbook/` 同性质）
- **角色**：被业务项目 `@import` 加载，提供"需求 → 上线"的可追踪自动化流水线
- **不是**：业务项目（不需要走 S1-S5、不需要 PRD 自己）
- **覆盖范围**：S2 末 → S3 → S4 → S5 的需求迭代循环（S1 方向探索仍走 gstack 八问）

## 2. 核心设计原则

### 2.1 人在环卡点设计（最关键决策）

| 环节 | 模式 | 主人介入度 |
|---|---|---|
| 环 1-5（分析 → PRD → 4A → 研发 → 测试） | **全自动**串联 | 0（产物随时可看但不卡流程） |
| 环 6（合入上线） | **🔴 硬卡点** | 主人确认才合入 |

主人体验设计：

1. 看需求总报告（一页 HTML 列出所有需求状态）
2. 点开任一需求**下钻看全链路**：原始需求 → 分析 → 设计 → 代码 → 测试，每环可追溯
3. 流程末尾点确认 / 打回完成合入

### 2.2 4A 架构 = 活文档

每条需求合入后，4A 增量自动并入主架构图。**4A 永远是最新版，不是一次性产物**。

- 业务架构（BA）：业务流程图、角色、业务能力
- 应用架构（AA）：产品架构图、功能特性清单
- 数据架构（DA）：数据架构图、数据流转图
- 技术架构（TA）：技术栈、部署架构

### 2.3 需求 ID 追踪链

每条需求一个唯一 ID（如 `R-2026-001`），所有产物用 ID 串起：

```
需求池/R-2026-001_用户登录/
├ 0_需求卡.md          # 原始需求
├ 1_分析报告.md         # 环 1 产出
├ 2_PRD.md / .html      # 环 2 产出
├ 3_4A增量.md           # 环 3 产出
├ 4_代码.md             # 环 4 产出（git 链接）
├ 5_测试报告.html        # 环 5 产出
├ 6_合入确认.md          # 🔴 主人改 status
└ status.yaml          # 当前状态
```

### 2.4 HTML 阅览模式

- 🟢 **持续渲染**：MD 一变 mkdocs 自动出，主人手机任何时候打开都是最新
- 🔴 **唯一硬卡点出独立合入报告 HTML**：含需求 / PRD / 代码 diff / 测试结果 / 4A 变更预览 五位一体

## 3. 六环节流程

```
新需求进入需求池
    ↓
[环 1] 需求分析 Agent 读取项目 4A 架构
    ↓
[环 2] 基于现状 gap，设计需求方案（PRD）
    ↓
[环 3] 4A 架构图同步更新（增量）
    ↓
[环 4] 研发 Agent 实施（接 superpowers TDD）
    ↓
[环 5] 测试 Agent 验证（接 Playwright Agents）
    ↓
[环 6] 🔴 主人体验确认 → 合并上线 → 4A 定稿
```

## 6. 与现有体系的关系

| 现有体系 | 与本框架的关系 |
|---|---|
| S1-S5 五段矩阵 | 本框架仅覆盖 S2 末 → S5；S1 不动 |
| gstack 思维底座 | 不动，新方向先过八问 |
| superpowers | 研发 Agent 通过 superpowers TDD / 调试 / worktrees 落地 |
| Playwright Agents | 测试 Agent 通过 Planner / Generator / Healer 三件套落地 |
| HTML 优先输出范式 v1.0 | 完全兼容，本框架是其落地实例 |
| mkdocs（`_view/`） | 持续渲染层，本框架接入项目自动得 |

## 9. 讨论备忘

### 主人原始要求（来自桌面草稿）

1. 原始需求汇聚到指定文件夹 → 设计为"需求池/" + 需求 ID 追踪链
2. Agent 自动分析需求 → 环 1（需求分析 Agent）
3. 基于本项目现状设计需求方案 → 4A 架构作为现状载体
4. 研发 Agent + 测试验证 → 环 4 + 环 5
5. 主人体验确认后合并 → 环 6 唯一硬卡点
6. 参考华为 4A 架构，持续迭代维护 HTML → 4A 活文档 + HTML 双轨阅览

### 奴婢小昭的关键判断（避免误读）

- **这是规则框架，不是业务项目**（避免一开始误读为业务项目立项）
- **人在环只需一个硬卡点**（避免设计过多卡点拖慢流程；主人原话："前面都全自动，只是最后合入功能的时候，人在环确认"）
- **4A 是活文档**（避免设计为一次性产物，跟主人"持续迭代维护"诉求一致）
- **v0.1 MVP 走"骨架 + 模板"**（避免一上来做 Agent 自动编排引擎，造大轮子）

### GitHub 调研结论（可借鉴模块）

| 项目 | 借鉴点 | 用在本框架哪里 |
|---|---|---|
| GitHub Spec Kit (90K⭐) | 阶段模板 + 产物结构 | 1_流程定义 / 3_产物模板 |
| Playwright Agents (官方 v1.56+) | Planner → Generator → Healer 三件套 | 4_Agent协议/测试Agent.md |
| BMAD-METHOD | 角色化 Agent + Story 打包 | 4_Agent协议（多 Agent 接力设计） |
| Agent-Flow | 过程可视化 + 日志重放 | 5_HTML渲染层/看板模板.html |

**整合思路**：骨架借 Spec Kit 阶段切分 → 角色补 BMAD 五角色 → 过程层接 Agent-Flow 可视化思路 → 测试段嵌 Playwright Agents 三件套，HTML 看板自建轻层。
