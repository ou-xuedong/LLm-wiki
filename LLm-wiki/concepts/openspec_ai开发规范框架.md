# 🧠 OpenSpec（AI 开发显式约束框架）

## 1. 定义与核心原理
OpenSpec 是一种将模糊的 AI 开发意图转化为**显式工程约束**的规范框架。其核心思想：把"工地门口跟工人说'你看着盖吧'"的 Prompt 方式，升级为建筑行业"施工蓝图"式的规范化表达。

**本质**：让 AI 从"盲写"变成"按图施工"。

## 2. 核心结构（五层规范）

| 层级 | 作用 | 示例 |
|---|---|---|
| **Goal（目标层）** | 定义模块要解决什么问题、成功标准、边界 | 目标：实现 JWT 用户登录认证 |
| **I/O（接口层）** | 输入输出格式、数据类型、错误码定义 | Input: `{username, password}` / Output: `{token, user_id}` |
| **Constraints（约束层）** | 性能要求、安全要求、禁止行为 | 密码必须前端加密、连续失败5次锁定30分钟 |
| **Style Guide（风格层）** | 命名规范、注释风格、文件结构 | 函数命名：动词+名词，统一使用 AppError 类 |
| **Modules（模块层）** | 功能拆分粒度、模块依赖关系 | 登录模块不包含注册、第三方登录 |

## 3. 关键价值
- **变更治理**：每一步变更都有迹可查，提案 → 归档 → 主规范
- **上下文持久化**：给 AI 一个可以读取的"记忆库"，解决多轮对话失忆问题
- **质量前置**：在写代码之前就把"成功标准"定义清楚

## 4. 操作工具
- `/opsx:propose` — 生成变更提案（变更意图 + 影响范围 + 设计方案）
- `/opsx:archive` — 将通过变更合并进主规范

## 5. 与主人方向的关联
主人做**亚马逊投流决策助手**，OpenSpec 逻辑可直接映射：
- **Goal** = 投流目标（预算/ACOS/销量）
- **I/O** = 关键词数据、竞品出价、ROI 指标
- **Constraints** = 预算上限、出价区间、禁投类目
- **Style Guide** = 策略命名规范、报告格式

**参考意义**：投流策略的"显式约束化"是 AI 投流助手能否稳定输出的核心前提。
