# 🔧 Harness 工程（Harness Engineering）

## 1. 定义与核心命题

**Harness**（挽具/羁绊）是 AI Agent 运行环境与外围基础设施的总称。

玄姐提出的核心命题：
> 一个 AI Agent 的真正智能上限，往往受限于它的 Harness（运行环境与外围基础设施），而非仅仅是底层模型。

AI产品不是"一种"产品，而是至少10种完全不同的产品范式。编程的核心正在从单纯依赖"模型参数规模"，转向对"上下文与意图的精准管理"。

## 2. Harness 的三层含义

### 2.1 狭义：工具链封装
将底层工具（搜索、文件读取、命令执行）封装为 LLM 可调用的标准接口。

典型工具链：
- ripgrep（代码搜索）
- Glob（文件路径匹配）
- FileRead（按行读取）
- Bash（命令执行）

### 2.2 中义：上下文管理机制
管理 Agent 与环境的上下文交互，包括：
- Prompt Cache（提示词缓存）
- Context Auto-compaction（上下文自动压缩）
- 会话上下文边界控制

### 2.3 广义：运行环境与基础设施
包括部署架构、安全机制、监控告警、权限控制等完整的企业级支撑体系。

## 3. Harness 设计模式（12 种）

文章提到 Martin Fowler 提出的 12 种 Agentic Harness 设计模式，核心包括：

| 模式 | 作用 |
|------|------|
| 工具链下放 | 底层工具直接暴露给 LLM |
| 探索即循环 | LLM 自主迭代搜索策略 |
| 软引导代替硬约束 | 不预设固定流水线，自主剥丝抽茧意图 |
| Sub-agent 隔离 | 派生只读子 Agent 处理大范围任务 |
| Context Auto-compaction | Token 逼近阈值时强制压缩 |
| Prompt Cache | 静态背景切块缓存，复用成本降低 80%+ |

## 4. 企业落地的核心痛点

### 4.1 Token 成本爆炸
多轮试错搜索倾倒无用内容 → 拖慢响应 + 产生天价 API 费用。

**解决路径**：
- Prompt Cache：重复前缀成本压降 80%+
- Context Auto-compaction：强制摘要压缩
- Sub-agent 隔离：中间日志自行消化，只回传精炼结论

### 4.2 索引维护成本
预建向量索引需要与实际代码保持同步，规模越大同步成本越高。

**解决路径**：在一定规模边界内，直接用高性能正则匹配替代向量索引。

## 5. 与"上下文图谱"的关系

Harness 工程的核心竞争点：**Context is King**。

- 嵌入式 AI（Copilot）：上下文是核心竞争力
- Agentic Search：上下文管理决定 Token 成本
- 记忆系统：结构化分层记忆（短期/长期/主动压缩）

## 6. 三条本质规律

**规律一**：AI产品的上限由模型能力决定，不是产品设计能力。
- 模型能做什么，产品边界就在哪里
- 要求 AI PM 对模型能力边界有真实认知

**规律二**：工作流深度，是比模型能力更可持续的护城河。
- 核心竞争力不是模型本身（模型可替换）
- 而是对特定工作流的深度嵌入和用户数据的持续积累
- "上下文护城河"比"模型能力护城河"稳得多

**规律三**：终局是 Agent，但演进是分阶段的，不要跳步。
- 短期：Copilot 类产品（嵌入增强）
- 中期：工作流自动化爆发
- 长期：高可靠性自主执行 Agent
