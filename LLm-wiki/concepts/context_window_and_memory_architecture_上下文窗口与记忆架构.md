# 🧠 上下文窗口与记忆架构

## 1. 定义与核心原理

**上下文窗口（Context Window）** 是 LLM 在一次推理中能"看见"的最大 token 数量，本质是 Transformer 架构中注意力机制的数学边界。

**记忆架构（Memory Architecture）** 是在上下文窗口之外，构建外部持久存储系统，让模型跨会话、跨任务保留关键信息。

两者解决的是同一个根本矛盾：**LLM 的注意力是有限的，但真实世界的任务需要无限的记忆。**

---

## 2. 关键技术路线与变体

### 2.1 上下文窗口扩展路线

| 方案 | 代表玩家 | 技术实现 |
|---|---|---|
| 暴力扩展原生窗口 | Gemini 100万token、Kimi 200万token | 更多注意力计算，更大显存需求 |
|稀疏注意力| GPT-4 / Claude | 不对所有 token两两计算，节省资源 |
|滑动窗口注意力| StreamingLLM | 保留局部窗口 + 关键锚点token |

### 2.2 外部记忆架构路线（核心）

| 方案 | 核心思路 | 优点 | 缺点 |
|---|---|---|---|
| **RAG（检索增强生成）** | 历史对话/文档存向量库，查询时召回注入上下文 | 成熟、实现简单 | 召回质量依赖检索算法 |
| **分层记忆架构** | 分 Working / Episodic / Semantic 三层，各层不同策略 | 类人记忆结构 | 设计复杂 |
| **递归摘要压缩** | 模型定期将历史压缩成摘要，保留精华 | 存储成本低 | 有信息损失，不可逆 |
| **外部状态读写** | 模型自行决定何时写入/读取外部存储（类MPC协议） | 模型自主决策 | 需要协议层支持 |
| **混合方案（主流）** | 最近对话走原生窗口 → 中期结构化DB → 长期向量库 | 性价比最高 | 需要各层协同设计 |

## 3. 行业应用场景

- **编程助手**：长时间任务中保留项目上下文（如 Claude Code 的 session 记忆）
- **客服机器人**：跨会话记住用户偏好和历史交互
- **AI Agent**：自主运行数小时/数天，需要持续记忆任务状态
- **投流助手**：宪法规则（长期记忆）+ 当前任务（工作记忆）+ 用户反馈（ episodic 记忆）

## 4. 代表性论文/项目

- **MemGPT**（UC Berkeley）— 分层记忆管理，类似操作系统的虚拟内存
- **ChatDB** — 用 SQL 数据库作为外部记忆存储
- **Anthropic MCP（Model Context Protocol）** — 模型自行读写外部工具和记忆的标准协议
- **Claude Code 的 AutoDream**（2026-03）— 自动合并、修剪跨会话记忆文件

---

## 5. 争议与开放问题

- **"无限上下文"是营销还是真实？** 上下文窗口再大也有物理边界，外部记忆是必经之路
- **召回质量 vs 召回速度**：向量检索延迟影响实时交互体验
- **记忆的真实性问题**：记忆是否会"幻觉"？如何验证召回内容的准确性？
- **隐私与安全**：外部存储的记忆数据如何防止泄露？
