# Agentic

# 🧠 Agentic Search（意图驱动搜索）

## 1. 定义与核心原理

**Agentic Search** 是一种由 LLM 驱动的动态搜索范式，核心是**放弃预建向量索引**，改用 LLM 自主决策搜索策略，通过多轮"探索→定位→验证→调整"的迭代循环完成任务。

与传统静态 RAG 的本质区别：
- **静态 RAG**：预先切块 → 计算 Embedding → 存入向量库 → 检索时比对向量距离
- **Agentic Search**：LLM 自主决定"搜什么关键词"、"用什么搜索模式"，按需动态探索

核心洞察：**代码中的类名、函数名本身就是工程师预埋的高精度锚点**，与其花算力猜测多维向量距离，不如让 LLM 自己拿起 Grep 精准出击。

## 2. 技术架构：零索引 Harness 系统

### Claude Code / Codex 的三层设计

**① 工具链下放**
将底层文本搜索工具（ripgrep、Glob、按行读取的 FileRead）封装为标准接口，直接暴露给 LLM 调用。

**② 探索即循环**
LLM 自主决定搜索关键词和搜索模式（仅看文件名 vs 看上下文），多轮迭代：
```
先定位 → 深入查看 → 结果不满意则调整策略 → 重新搜索
```

**③ 软引导代替硬约束**
不预设固定的"先检索后生成"流水线，让模型在多轮交互中完成对代码意图（Intent）的剥丝抽茧。

### 底层性能支撑

现代 ripgrep 的极致性能让"零索引"成为可能：
- **五层漏斗过滤**：遵守 .gitignore 剪枝，跳过二进制文件，瞬间收敛扫描范围
- **SIMD 向量化加速**：CPU 字节级并行比较
- **mmap 零拷贝**：大文件内存映射读取
- **Page Cache 命中**：本地项目常驻操作系统内存缓存

实测：百万行级别代码库并发正则扫描 < 100ms。

## 3. 两大范式对比

| 维度 | 零索引 + Grep（Claude Code 模式） | 混合双索引（Cursor 模式） |
|------|-----------------------------------|--------------------------|
| 索引方式 | 无预建索引，运行时动态搜索 | 语义 Embedding + Trigram 倒排 |
| 适用规模 | MB ~ 数百 MB（中小型代码库） | GB 级别（超大型代码仓库） |
| 启动延迟 | 零 | 需要离线索引构建 |
| 维护成本 | 零 | 索引与代码同步维护 |
| 响应时间 | < 100ms（Page Cache 命中） | 稳定可预测 |
| 适用场景 | 本地项目研发、个人/团队 | 百人协作、跨项目知识域 |

**结论**：两者不是优劣之争，是业务规模与场景的取舍。

## 4. Token 成本与 Context 管控

纯 Grep 方案的最大隐患：**Token 爆炸**。多轮试错搜索极易将无用代码片段倾倒进 Context Window。

三大自我收敛机制：

### 4.1 Prompt Cache（提示词缓存）
将 System Prompt 等静态背景切块并利用 API 缓存机制，重复计算成本压降 **80%+**。

### 4.2 Context Auto-compaction（上下文自动压缩）
当累计 Token 逼近阈值时，强制触发摘要生成机制，将早期搜索试错轨迹压缩为精炼结论，为后续操作腾出空间。

### 4.3 Sub-agent 隔离机制
遇到大范围搜索任务时，主控 Agent 派生只读"Explore 子 Agent"：
- 子 Agent 在独立上下文中产生大量中间检索日志（Grep/Read 结果）自行消化
- 最终只向主进程返回一段**高信息密度的结论**
- 主进程不被中间过程污染

## 5. 核心竞争壁垒分析

| 壁垒类型 | 评估 |
|---------|------|
| 技术壁垒 | 中（现代搜索工具已开源，ripgrep 等性能极致） |
| 工程壁垒 | 高（Harness 系统设计需要深度调优） |
| 数据壁垒 | 低（不依赖预建索引） |
| 网络效应 | 低（工具链标准化后易复制） |

## 6. 对主人方向（投流决策助手 Agent）的映射价值

- **Sub-agent 隔离** → 投流多账号管理：主控 Agent 派生子 Agent 处理单个账号，相互隔离
- **Context 压缩** → 长对话历史总结，腾出空间给新任务
- **Token 成本管控** → 投流决策场景同样面临 API 调用成本优化问题
