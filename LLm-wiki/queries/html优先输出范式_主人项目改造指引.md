> **本文用途**：当主人想把任何一个项目（卟卟鸡 / 知识库 / 方向探索 / 未来新项目）的 AI 输出从 Markdown 升级为 HTML 时，搜出这一篇就能照单执行。**这是改造手册，不是论文**。

## 1. 缘起与核心论断

### 1.1 缘起
- **来源**：Anthropic Claude Code 团队成员 Thariq 文章《抛弃 Markdown！使用 HTML 和 Agent 交流效果好到离谱》（2026-05-09，AI寒武纪转载）
- **作者主张**：AI 时代默认输出格式应从 Markdown 切换为 HTML，作者本人已"完全停用 Markdown"
- **配套案例库**：https://thariqs.github.io/html-effectiveness/
- **本文修正**：Thariq 主张"全面 HTML 化"，奴婢实测后做出**分层修正**——不是所有文件都该 HTML 化，关键看"读者是谁"

### 1.2 核心论断（三句话）
1. **Markdown 的优势已死**：作者原本说"易于人编辑"，但 AI 时代 99% 的 MD 文件是 AI 写 AI 改，这条优势消失；超过 100 行的 MD 文件没人读
2. **HTML 的优势爆炸**：信息密度（表格/SVG/CSS）+ 视觉层级（标签页/响应式）+ 交互（滑块/复制按钮）+ 分享（一个链接搞定）+ 数据吞吐（Claude Code 遍历文件夹生成总览页）
3. **决定一份文件该不该 HTML 化的根本判据**：**读者是 AI 还是主人？**

### 1.3 核心金句（v1.2 沉淀 · 来自新智元报道）

- ✨ **"当瓶颈从 token 数量变成了人的注意力，格式的选择逻辑就彻底反转了"** —— 解答 HTML 多耗 2-4 倍 token 是否值得的根本判据：值得，因为人会读完 HTML，不会读完长 MD
- ✨ **"读了 vs 没读，这个差距比 token 效率大"** —— 同上
- ✨ **"代码 diff 是空间信息，Markdown 把它压平了，HTML 能把它还原回来"** —— 为何 PR 审查必须 HTML 的根本论据
- ✨ **"AI 生成的不再是文档，而是工具"** —— Throwaway Editor 的本质

## 2. 分层模型：AI 喂养层 vs 主人消费层（核心心法）

> 这是本指引最关键的一刀。Thariq 没讲透，但所有改造都建立在这里。

### 2.1 AI 喂养层（保留 Markdown）
**特征**：给 Agent 自己读的指令、规范、模板、日志
- CLAUDE.md / 各级 README / schema 文件
- 模板、规则、工作流定义
- 反馈日志、提交记录、log.md
- 知识库的实体卡片、概念卡片（Agent 检索后做综合分析）

**为什么不改 HTML**：
- AI 读 MD 比 HTML 更高效（Token 消耗低）
- git diff 友好，主人手改也方便
- 这些文件本来就不是用来"看视觉"的

### 2.2 主人消费层（升级为 HTML）
**特征**：主人最终要读、要决策、要分享的产物
- 数据密集型报告（带表格/图表/趋势）
- 多视角对比（并排卡片网格）
- 决策类输出（带交互调参/一键复制）
- 复盘类报告（差异对比/颜色编码）
- 给团队/合作方看的总结

**为什么改 HTML**：
- 主人是肉眼读，HTML 视觉信息密度高 5-10 倍
- 可以加交互（滑块调仓、按钮复制提示词回 Claude Code）
- 双击浏览器打开即可，传 S3 后甩链接给团队
- 数据可用 Chart.js / ECharts / SVG 做可视化

### 2.3 边界判据卡片

| 你写的这份文件 | 谁会读它？ | 选什么 |
|---|---|---|
| Agent 工作规范 / 模板 / schema | AI 自己读 | **MD** |
| 主人交付的研判报告 / 复盘报告 | 主人 + 可能要发给老板 | **HTML** |
| 数据少的笔记 / 备忘 / 待办 | 主人速读 | **MD** |
| 含 5+ 表格 / 走势图 / 对比矩阵 | 主人深度研判 | **HTML** |
| Agent 间传递的中间产物 | 下一个 Agent | **MD/JSON** |
| 给非技术人看的成果 | 同事 / 老板 / 客户 | **HTML** |

## 3. 三问决策树（套到任何文件上）

```
┌─ Q1：主要读者是谁？
│   ├─ AI Agent → MD（结束）
│   └─ 主人 / 第三方 → 进 Q2
│
├─ Q2：信息密度高吗？（5+ 表格 / 多维数据 / 趋势）
│   ├─ 否 → MD（结束）
│   └─ 是 → 进 Q3
│
└─ Q3：需要交互或分享吗？（滑块 / 按钮 / 链接传播）
    ├─ 否 → MD 嵌入式（MD + Mermaid + 内联 SVG）
    └─ 是 → 纯 HTML
```

## 3.5 四模式辨析（v1.1 升级 · 2026-05-09 晚）

> 主人 5/9 晚反向追问"渲染 vs 同时生成"的概念混淆——奴婢梳理为四模式。本节是对 §3 决策树的"实施层"补充：决策树解决"该不该 HTML"，本节解决"HTML 怎么落地"。

| 模式 | 名字 | 谁是真相源 | 改 MD 后 HTML 表现 | 工具 | 适用场景 |
|---|---|---|---|---|---|
| **1** | 渲染层（MkDocs/Docsify） | MD 唯一源 | 立即可见 | `mkdocs serve` 起本地服务 | **AI 喂养层 + 历史档案的视觉化阅览** |
| **2** | 双轨同时生成 | MD + HTML 双源 | HTML 不自动更新 | 自建钩子 | ❌ 反模式（双源失同步、维护噩梦） |
| **3** | 替换（Thariq 流派） | HTML 是产物，无对应 MD | N/A | Claude 直接生成 | **主人消费层报告类产物** |
| **4** | 编译（YAML→HTML） | YAML/JSON 是源 | 重跑脚本 | 渲染脚本 | 第二阶段；HTML 退化为渲染产物 |

**关键判定：**
- 同一个项目**可以同时存在多种模式**：卟卟鸡的 CLAUDE/专家智库走模式 1（mkdocs 阅览），评估记录走模式 3（Claude 生成），未来评测脚本可走模式 4（YAML 编译）
- **模式 2 是反模式**：必然双源失同步；主人最初提"每个 MD 都加 HTML"即此模式，已劝阻
- **模式 1 与模式 3 互补**：模式 1 解决"想方便读所有 MD"，模式 3 解决"特定报告类产出值得视觉升级"——二者非此即彼是误判

## 3.6 三种 HTML 新形态（v1.2 升级 · 2026-05-10）

> 来源：新智元《Claude Code 隐藏玩法爆火！Anthropic 大佬：不要再用 Markdown 了》（2026-05-09）。在 §4 五场景对位的基础上，新增三类**强交互/强结构性**的 HTML 形态。

### 3.6.1 Throwaway Editor（用完即弃的编辑器）

**定义**：为当前任务量身定制的单文件 HTML 工具，用完即扔，**不是产品、不复用、不保存状态**。

**核心闭环**：人 → AI 生成 HTML → 人在 HTML 里操作（拖拽 / 调滑块 / 选择）→ 一键导出 → 把结果粘回 AI 继续干活。

**关键论断**："**AI 生成的不再是文档，而是工具**"。

**典型形态**：
- N 张可拖拽卡片 → 分到「Now / Next / Later / Cut」四列 → 一键导出 markdown
- 左编辑右预览（Prompt Tuner 形态）→ 调好参数一键复制
- 滑块调参（动画曲线 / 仓位 / 阈值）→ 实时预览 + 复制按钮

**对位主人项目**：
| 项目 | 适配场景 |
|---|---|
| 卟卟鸡 | 调仓器（4ETF 拖到「持有/移止损/减半/清仓」）/ 板块优先级排序 / 标的入选剔除 |
| 方向探索 | 多方向 priority 拖拽分级 / 战略模式四象限拖拽决策 |
| 知识库 | 实体批量重排 / 标签批量分组 |
| 产品研发型（S1） | 候选方案评分卡拖拽 / 风险登记册严重度调整 |

**5/10 试点交付物**：`卟卟鸡投资决策/模板/_DEMO_4ETF调仓器_Throwaway.html`

### 3.6.2 Risk Map 风险地图（代码审查 HTML）

**定义**：PR 审查 HTML——顶部一张风险地图（被改文件按风险颜色编码），旁边内联注释（红 Blocking / 黄 Worth a look / 绿 Safe），底部自动生成修复建议清单。

**关键论断**："**代码 diff 是空间信息，Markdown 把它压平了，HTML 能把它还原回来**"。

**对位主人项目**：
- 产品研发型 S3 Code Review（亚马逊投流 / 算力 / 知识自进化引擎）
- 替代 GitHub 默认 diff 视图——给主人评审 + 给团队成员浏览
- 已纳入 `_shared/产品研发型.md` §1.5.1 的 S3 行"Sprint 末评审 HTML / 架构图 HTML 快照"

### 3.6.3 交互式概念教程

**定义**：把抽象概念做成**可点击、可拖拽、实时反馈**的单文件 HTML（如一致性哈希环 + 加节点/删节点按钮 + 实时 key 重分配）。

**对位主人项目**：
- 知识库 `wiki/concepts/` 的复杂概念卡——主人深读时按需生成 HTML 配套教程
- 不强求每张概念卡都做（326 张卡范围太大）
- 触发：主人说"我想搞懂 XX 是怎么工作的"时按需生成

**反模式**：不要给所有概念卡做交互式 HTML——按需生成，用完保留为可分享资产。

## 5. 改造 Playbook（7 步标准流程）

### Step 1：勘查现有产物
- `ls` 项目目录，列出所有"主人最终消费"的输出文件类型
- 抽样读 3-5 份，统计每份的"信息密度等级"（低/中/高）
- 标记哪些超过 100 行 = 主人读起来痛苦的重灾区

### Step 2：分层切分（最关键的一步）
- 用 §2 的判据卡片，给每类文件贴 `MD` / `HTML` / `MD+嵌入式` 标签
- **铁律**：CLAUDE.md / schema / 模板 / log 一律保 MD

### Step 3：选试点
- 从主人消费层挑 **1 个**最高频、最数据密集、最有视觉提升空间的产物作首发
- 不要一开始就全改——先验证主人审美

### Step 4：建模板（设计系统）
- 在项目根目录新建 `模板/` 目录
- 写一份 `report_template.html` 作为视觉基线
- 推荐栈：原生 HTML + Tailwind CDN + Chart.js CDN（零依赖、可双击打开）
- 颜色规范：看多/看空/中性 = 红/绿/灰；信心度用进度条；严重程度用颜色编码
- **审美一致性技巧**（Thariq 原话）：让 Claude 读你的代码库，先生成专属"设计系统 HTML 文件"，后续生成全部以它为参考

### Step 5：建工具链（可选，先静态后脚本）
- **第一阶段**：手工/Claude 直接生成静态 HTML，先验证设计
- **第二阶段**：写 `工具/render_report.py`，从结构化数据（YAML/JSON）渲染 HTML
- **不要一上来就建脚手架**——先静态原型主人验收后再投入

### Step 6：增 CLAUDE.md 条款
- 在项目 CLAUDE.md 新增"输出格式分层"小节
- 明确：哪些产出强制 HTML、哪些保 MD、模板路径在哪、视觉规范是什么
- 写明触发：主人说"出报告""做复盘""生成方案"时默认走 HTML

### Step 7：保鲜与同步
- 知识库类：`qmd embed --collection wiki`
- git 同步：`bash ~/Desktop/AILab/sync.sh push`
- 后续每次产出新 HTML 报告，记得更新模板（如有视觉迭代）

## 7. 陷阱与权衡

### 7.1 已知短板（来自 Thariq + 奴婢实测）
| 短板 | 缓解方案 |
|---|---|
| 费 Token（HTML 比 MD 多 2-4 倍） | Opus 4.7 百万级上下文窗口下基本无感；只在主人消费层用 |
| 生成慢 2-4 倍 | 接受这个代价；视觉收益远大于等待成本 |
| git diff 杂乱（HTML 难审查） | 把 HTML 视为"产物"而非"代码"；用结构化数据（YAML）做 source，HTML 是渲染产物 |
| 审美参差 | 项目第一份 HTML 时建立"设计系统 HTML"，后续全参照 |
| 多设备适配 | 模板默认带响应式 + Tailwind |

### 7.2 反模式（不要做）
- ❌ 把 CLAUDE.md / schema 改成 HTML（AI 喂养层不该改）
- ❌ 一上来就写复杂渲染脚本（先静态原型验收审美）
- ❌ 每份报告各搞各的视觉风格（必须用统一模板）
- ❌ HTML 里嵌入大量主人要手改的纯文本字段（编辑成本反而更高）
- ❌ 把"qmd 检索的语料库"全 HTML 化（污染向量、提高 token 成本、收益为零）

### 7.3 何时回退到 MD
- 内容简单（< 50 行）
- 主要给 AI 读
- 需要频繁手改
- 需要 git diff 跟踪每行改动

## 11. 实操经验沉淀（v1.1 · 2026-05-09 晚）

### 11.1 卟卟鸡两种 HTML 形态确认

经 5/9 晚实证，卟卟鸡需要至少两类 HTML 模板：

- **研判报告**（`模板/研判报告_原型.html`）：首次买入决策——市场仪表盘 / MACD 多周期 / 7 专家会诊 / 操作方案 / 风险
- **盯盘指南**（`模板/_DEMO_4ETF盯盘_HTML样品.html`）：持仓后管理——整体仪表盘 / 持仓基本盘表 / 大盘背景 / 各 ETF 自身 MACD 卡片 / 止损止盈金字塔 / 大盘联动 / 盯盘三件事

未来可扩展：复盘报告、评测报告各自独立形态。

### 11.2 AILab 全局 MkDocs 阅览层（_view/）

**部署位置**：`~/Desktop/AILab/_view/`（已加根 `.gitignore`，不入任何项目仓）

**启动**：

```bash
cd ~/Desktop/AILab/_view && \
  ~/Library/Python/3.9/bin/mkdocs serve --dev-addr=127.0.0.1:8765
```

**架构**：`mkdocs.yml` + `docs/` 目录用 symlink 链入各项目（避免文件复制）。

**踩坑（已解决）**：
- `知识库/raw/Recent_KOL_Updates.md` GBK 编码触发 `UnicodeDecodeError: byte 0xbd`。按精准手术原则**不动主人源文件**，加 `exclude_docs: '**/raw/**'` 排除即可
- 跨项目相对链接（如 `../_playbook/xxx.md`）报 warning，不影响视觉化阅览

### 11.3 Thariq 真实实现反向印证

扒源码后发现 Thariq 是「**零依赖单文件 HTML**」派——Birchline 设计系统（CSS 变量 + 系统字体栈 + 内联 vanilla JS，**无任何 CDN**），与主人卟卟鸡当前模板（Tailwind CDN + Chart.js CDN）实现哲学有差异。

| 维度 | Thariq Birchline 派 | 卟卟鸡当前模板 |
|---|---|---|
| CSS 框架 | CSS 变量 + 原生选择器 | Tailwind CDN |
| 图表 | 内联 SVG / CSS 渐变 | Chart.js CDN |
| 配色 | 米白+深炭+橙红+燕麦+橄榄（Anthropic 风） | A 股红涨绿跌（领域专属） |
| 文件大小 | 15-21 KB 真自包含 | ~22 KB 自包含 + CDN |
| 离线可用 | ✅ 100% | ⚠️ 依赖 CDN |
| 设计系统 | 命名"Birchline"统一 tokens | 未命名，用 Tailwind 工具类 |

**主人当前选择的合理性**：A 股红涨绿跌符合用户认知；Tailwind 让排版调整快；当前无离线场景。

**未来若审美迭代可考虑**：给卟卟鸡命名专属设计系统（如"ETFstack"）+ 可选 Tailwind→CSS 变量化（提升离线 + 减少 diff 抖动）。**不强求与 Thariq 同色**——A 股语境下红涨绿跌正确。

### 11.4 §3.5 四模式实战搭配

5/9 晚实证：**模式 1 + 模式 3 互补搭配最优**——

| 文件类型 | 模式 | 阅览方式 |
|---|---|---|
| CLAUDE.md / 专家智库 / 决策流程定义 | 模式 1 | `_view/` MkDocs 视觉化 |
| 评估记录 / 复盘报告 / 评测报告 | 模式 3 | 浏览器双击 `.html` |
| 历史 MD 档案（如 5/6 旧盯盘指南） | 模式 1 | `_view/` MkDocs 视觉化 |
| 模板 HTML | 不入站 | 浏览器双击 |
| schema / log / 反馈日志 | 模式 1 | `_view/` MkDocs |

**主人最初疑问的最终答案**：不存在"加 HTML 给所有 MD"的必要工作量——AI 喂养层模式 1 视觉化即可，主人消费层走模式 3 即可，二者互补不重叠。
