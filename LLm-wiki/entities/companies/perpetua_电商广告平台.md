# Perpetua 电商广告平台

## 1. 核心定位与业务模式

Perpetua 是一个**目标驱动的电商广告自动化平台**，核心理念是"用户定义战略目标（增长/盈利/品牌防御/认知度），AI 引擎自动执行战术"。成立于 2018 年，总部位于加拿大温哥华，在 Toronto、London、São Paulo、Berlin 设有办公室。累计融资超 3000 万美元（投资方包括 Ignia Partners），规模约 50-100 人。

G2 连续 9 次被评为 Leader，Amazon Advanced Partner。

**变现方式：** 订阅制（Essentials $695/月起）+ 广告 spend 抽成（Growth/Premium 层级）。

## 2. 护城河 (Moat)

- **AI 目标引擎**：用户设定目标后引擎全自动执行，无需手动创建 Campaign
- **小时级 intraday 优化**：接入 Amazon Marketing Stream，分钟级响应竞价变化
- **多市场多平台**：支持 Amazon、Walmart 等多个 marketplace
- **品牌心智**：G2 Leader + Amazon Advanced Partner 认证，在海外卖家中有强品牌认知
- **规模化客户基础**：服务大量中大型品牌、规模化 FBA 卖家、CPG 公司
- **强化学习技术壁垒**：强化学习 + LSTM/Transformer 模型，分钟级数据处理，形成决策反馈闭环

## 3. 命门与软肋 (Weaknesses)

- **定价门槛高**：最低 $695/月，中小卖家难以承受
- **黑盒问题**：AI 引擎执行战术，对想深度把控细节的用户不够透明
- **过度依赖 Amazon 生态**：多平台扩展能力相对有限
- **中小客户支持弱**：Growth 以上才有客户成功团队

## 4. 核心能力

- AI 驱动竞价算法（conversion-based bidding）
- 自动关键词采集与扩展（automated keyword harvesting）
- 目标导向型 Campaign 创建
- 小时级报表 + intraday 优化（Growth 层级）
- 小时级 Share-of-Voice 和自然排名监控
- 竞争对手搜索词数据 + 收入机会分析（Growth+）
- AMC 自定义报表（Premium）
- 支持 Sponsored Products / Sponsored Brands / Display
- **生成式 AI 广告文案创意**（2025 年引入）

### 5.2 用户目标 → 策略 → 战术 → 执行：完整链路

```
用户输入：战略目标（增长/盈利/品牌防御） + 约束（ROAS 500% / ACoS 20%）
                │
                ▼
【策略层 Strategy】
  Perpetua 将用户目标转化为 Amazon 竞价策略配置：
  • 增长 → 动态竞价-向上向下（Dynamic bid - up and down）
  • 盈利 → 动态竞价-仅向下（Dynamic bid - down only）
  • 品牌防御 → 固定竞价 + 手动位置溢价
                │
                ▼
【战术层 Tactics】
  引擎决定：在哪个广告位出多少钱、哪个词暂停、哪个词加码
  • 首页首位（Top of Search）：愿意付 1.5-3x 加价
  • 商品页面（Product Page）：维持基准出价
  • 转化率预测高的词：提高竞价
  • 长期无转化词：降低出价或暂停
                │
                ▼
【执行层 Execution】
  调用 Amazon Ads API 批量更新：
  • PUT /sp/campaigns/{campaignId}/bids
  • POST /sp/keywords/{keywordId}/bid
  • DELETE /sp/keywords/{keywordId}
  频率：每 5 分钟（Growth 层级）/ 每小时（Essentials 层级）
```

### 5.3 核心决策逻辑：单次竞价决策

**对于每一个广告展示机会，算法决策流程**：

```
Step 1：接收信号
  • 搜索词：wireless earbuds bluetooth
  • 用户出价：$1.00
  • 竞品数量：8 个
  • 历史 CTR：4.2%
  • 历史 CVR：9%

Step 2：计算预测价值
  预测销售额 = $1.00(竞价) × (1 + 加价因子) × CVR(9%) × 客单价($45)
             = $1.00 × 1.5 × 9% × $45
             = $6.075（预测GMV贡献）

Step 3：判断是否值得参与
  IF 预测销售额 / 竞价 ≥ 目标ROAS（5:1）
  THEN 参与竞拍，出价 = 略高于当前最高竞价的最小值
  ELSE 放弃此次展示

Step 4：竞拍结果
  胜出 → 展示广告 → 记录 impression
  失败 → 不消耗预算 → 进入下一轮
```

**简化核心公式**：

```
ACoS = 广告费 / 销售额
ROAS = 销售额 / 广告费 = 1 / ACoS

目标：在每日预算约束下，最大化 ROAS 达成率
约束：ACoS ≤ 用户设定上限
约束：花费 ≤ 每日预算
```

### 5.4 Intraday 优化循环（分钟级响应）

这是 Perpetua 区别于传统手动操作的核心能力：

```
【T=0 分钟】
  启动 Campaign，当前竞价配置：
  • 关键词 "bluetooth earbuds"：$1.00
  • 每日预算：$500

【T=5 分钟】
  数据拉取：Marketing Stream 最新 ACoS = 35%（目标 25%）
  动作：自动将竞价从 $1.00 → $0.80（降低 20%，单次最大调整幅度）

【T=30 分钟】
  数据拉取：关键词 "wireless earbuds" ROAS 突然飙升至 8:1
  动作：自动将竞价从 $0.80 → $1.30（加价 62.5%，快速占位）

【T=60 分钟】
  数据拉取：日预算消耗 $300，ACoS 21%（接近目标）
  动作：自动提高高效词竞价，同时降低低效词竞价，预算向高效词集中

【T=180 分钟】
  触发关键词 Harvest：新发现搜索词 "tws earbuds" 有 3 个订单
  动作：自动添加到投放词列表，以较低竞价开始测试

核心规则：
  • 单次竞价调整幅度 ≤ 20%（避免价格震荡）
  • 调整频率：每 5 分钟检查一次
  • 方向判断：基于 30 分钟滑动窗口的 ROAS 趋势
```

### 5.6 预算智能分配逻辑

```
每日预算 $500 在多个 Campaign 间的分配算法：

Step 1：识别高效 Campaign
  Campaign A：ROAS 6:1 → 高效 → 分配更多预算
  Campaign B：ROAS 2:1 → 低效 → 削减预算

Step 2：预算再分配
  原分配：A=$200, B=$200, C=$100
  优化后：A=$280 (+40%), B=$120 (-40%), C=$100 (不变)

Step 3：实时响应
  若 A 在上午就消耗完日预算的 80%
  → 自动从 B 临时调配预算给 A（小时级调整）

Step 4：跨日学习
  每天结束后，将效果数据喂入 LSTM 模型
  → 预测次日每个 Campaign 的最优预算分配
```

### 5.7 API 调用模式

```
数据拉取（感知层）：
  • Reporting API：每 5 分钟拉取最新报表（/reports/sp/...）
  • Marketing Stream：Amazon Marketing Stream 实时推送（每 15 分钟）
  • SP-API Catalog：竞品 ASIN 数据（每小时）

数据写入（执行层）：
  • Sponsored Products API：批量更新竞价（PUT /bids）
  • Keyword API：增删关键词（POST /keywords, DELETE /keywords）
  • Campaign API：调整预算（PUT /campaigns/{id}/budget）

API 调用特征：
  • 批量操作：每次最多 100 个关键词
  • 速率限制：遵守 SP-API 2请求/秒上限
  • 重试机制：429 时指数退避
  • 幂等设计：重复调用不会重复扣费
```

### 5.8 与主人投流决策助手的关系

| Perpetua 能力 | 对主人方向的启发 |
|--------------|----------------|
| 分钟级 intraday 优化 | 主人产品需要同样的实时感知→决策闭环 |
| 强化学习探索最优竞价 | 主人壁垒在于**决策智能**，而非 API 调用 |
| 关键词 Harvesting | 主人可做的差异化：比 Perpetua 更懂**选词+归因** |
| 目标驱动（用户设 ROAS） | 主人可做的差异化：**帮用户设对目标**，而非交给用户设定 |
| 全自动"设置后忽略" | 主人方向：**决策建议**而非全自动执行，保留用户控制感 |
| 黑盒不透明 | 主人机会：**可解释的决策理由**，让用户理解"为什么要这样投" |

**核心结论**：Perpetua 证明了"投流自动化"可行，但其全托管模式对想保持控制感的 SMB 用户有隔阂。主人方向用"决策副驾"而非"代驾"切入，既能复用其 API 架构逻辑，又能避开直接竞争。
